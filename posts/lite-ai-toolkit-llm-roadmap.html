给 lite.ai.toolkit 加 LLM 推理：路线 + 待办

2026.05.09 LLM TensorRT OSS Roadmap

这篇不是一篇文章，是一份给我自己看的项目跟踪表。目标：在 xlite-dev/lite.ai.toolkit 这个 4.4k stars 的 C++ TRT 工具包里 加一个 LLM 推理子模块。下面的 checkbox 状态会被存在本地浏览器里， 所以下次打开这一页，已经勾过的还在。 每个阶段标题后会自动显示进度，例如 3 / 7。

目标（3–6 个月）

主目标

  • 在 lite.ai.toolkit 仓库里有一个被 maintainer 合并的 LLM 子模块（lite/trt/llm/），至少包含一个完整可运行的模型（计划是 Qwen2.5-0.5B）
  • 配套 5 到 8 篇带 benchmark 数据的博客文章，按现有"慢慢写"的写法
  • 3080 Ti 上至少跑通：FP16 baseline、KV cache 重用、CUDA Graph、INT8 weight-only 量化 这四组对比数字

衍生目标

  • 简历 / GitHub profile 上能写"4.4k stars OSS 项目核心模块贡献者"
  • 长期定位从"CV TRT 加速"转向"CV 加速 + 多模态 LLM 推理"，差异化避免和大盘子卷
  • 积累一份可以反复在新模型上复用的 LLM 推理工程化模板（KV cache、采样器、ONNX 导出脚本、benchmark 框架）

不在范围内（显式 anti-scope）

这一节的优先级高于待办清单本身。最常见的失败模式是做着做着范围漂移，所以把不做的事情列出来。

  • 不做 vLLM / TRT-LLM 的平替——这是错位的目标，做不完
  • 不做训练相关（包括 LoRA / fine-tuning / RLHF）
  • 不做 7B 以上的模型（3080 Ti 12GB 显存做不动 FP16；INT8 量化的 7B 是上限）
  • 不引入新的第三方依赖——尊重 toolkit 的"lite"定位，第一版 tokenizer 自己写，不引入 sentencepiece / tokenizers-cpp
  • 不做 multi-GPU / tensor parallel / pipeline parallel
  • 不做服务化框架（HTTP server、scheduler、batch manager），只做单进程推理库
  • 不做 FP8 量化——3080 Ti 不支持，等以后换卡

能力探针：先用 2 周验证我能不能走

这一阶段的目标不是产出代码，是验证"我有没有能力把这条路走完"。 数据出来再决定要不要 commit 后续 phase。

判断标准：以上 8 条至少完成 6 条，且没有任何一条让我"完全卡住、看不到出路"。 满足就进 Phase 0；不满足就回头看具体是哪一条卡住，针对性补，而不是放弃整条路线。

Phase 0：发 RFC issue（1 天）

Phase 1：最小可跑模块（6–9 周）

PR1：Tokenizer 抽象 + GPT2-style BPE（1.5 周）

PR2：lite/trt/llm/ 骨架 + KVCacheBuffer + Sampler（2–3 周）

PR3：trt_qwen2.{h,cpp} + example，端到端跑通（2–3 周）

PR4（软实力，可选）：把 SD 模块也重构进 BasicTRTHandler（1 周）

Phase 2：优化层，每条一个 PR + 一篇博客

这一阶段的颗粒度比 Phase 1 粗。每一项都既是代码也是博客素材，按需挑选优先级。

Phase 3：多模态推理（差异化卖点）

这是整个 roadmap 的差异化部分：纯 LLM 加速的人多，但能把 toolkit 已有的 vision encoder 和新加的 LLM 拼起来做高性能多模态推理的人少。

风险 / 失败模式

  • 心理上：4 周后觉得太慢就放弃。预期对齐到 3-6 个月才合理；如果第 4 周开始焦虑，回头看本表的进度而不是看绝对时间
  • 技术上：ONNX 导出 LLM 卡 1-2 周。这是已知的坑，单独排过程做成一篇博客，把"踩坑"本身当成产出
  • 协作上：owner 拒绝方向。Phase 0 的 RFC 必不可少，不要默默写完一个月才提交
  • 范围漂移：开始想做 vLLM 平替。每周回头读一遍"不在范围内"那一节
  • 第一个 PR 被晾着。如果 PR1 提交后两周没反馈，礼貌地在 issue 里 ping，再不行就把 PR1-3 一起提交，一次性给 maintainer 评估
