llama2.c 精读指引：只读这 5 段，为写 LLM TRT 模块备课

2026.05.09 llama2.c LLM KV cache Reading

这是 给 lite.ai.toolkit 加 LLM 推理路线图 的配套备课材料。karpathy/llama2.c 共 700 行 C 代码，但你不需要全读。 目标只有一个：读完之后能在白纸上画出 KV cache 的内存布局， 能解释 prefill 和 decode 阶段为什么要分开。其他都是次要的。 预算 1.5 小时，专注读这 5 段就够。

为什么是这个文件

市面上能完整跑通的 LLM 推理实现里，karpathy/llama2.c 是最适合"备课"的一个：

  • 700 行，单文件，无任何外部依赖
  • 包含完整的 Llama 2 transformer 前向传播 + KV cache + 采样 + 生成循环
  • Karpathy 写的，命名清楚、不耍花招、注释合理
  • 结构和你将来要在 TRT 里"指挥"的东西一一对应，但因为是 CPU 实现，每个细节都看得见摸得着

你不是要 port 它——TRT engine 会替你处理 90% 的矩阵运算。 你是要从它身上看清"那 10% 你必须自己写的胶水代码长什么样"： KV cache 的布局、prefill / decode 循环、采样器、EOS 检测。

开始之前的准备

  1. git clone https://github.com/karpathy/llama2.c
  2. 核心文件是 run.c，就读这一个文件
  3. 用一支笔，准备一张白纸——下面有几处需要你画

读完这一篇你应该能做的 4 件事

这是判断"读到位了没"的标准。读之前先记住：

  1. 能在白纸上画出 KV cache 在内存里的布局，标出 layer / seq position / head / dim 四个维度
  2. 能用一句话解释为什么 prefill 阶段输入是整段 prompt、decode 阶段每次只输入一个 token
  3. 能指出在每一步 decode 里，哪些张量是"要复用前面计算结果的"、哪些是"每步重新算的"
  4. 能说出从 logits 到下一个 token id 的完整采样流程（temperature → softmax → top-p → sample）

必读的 5 段（按顺序读）

Section 1 · 15 min · search: typedef struct

1. Config 和 TransformerWeights 这两个 struct

从文件开头开始，找到 Config 和 TransformerWeights 这两个 struct 定义。 这是整个模型在内存里的"零件清单"。

要看的：

  • Config 里的 dim、hidden_dim、n_layers、n_heads、n_kv_heads、vocab_size、seq_len 各是什么意思（在你旁边标注一下）
  • TransformerWeights 里每一项都是哪一层的哪一种权重——例如 wq 是 query 投影矩阵，w1/w2/w3 是 FFN 三个矩阵

动作：在白纸上把 transformer 一层的零件画出来：embed → norm → q/k/v 投影 → attention → o 投影 → norm → FFN(w1, w2, w3) → 残差。这张图就是后面 forward() 函数的执行流程。

Section 2 · 15 min · search: RunState

2. RunState struct，重点看 key_cache 和 value_cache 字段

RunState 是推理时的"工作台"，所有中间张量都在这。 你只需要专注两个字段：key_cache 和 value_cache。

要看的：

  • 这两个字段的类型（一维 float 数组）和分配尺寸（n_layers * seq_len * kv_dim 一类的乘积）
  • 那些 n_layers、seq_len、kv_dim 是怎么算出来的

动作：在白纸上画一个三维的盒子——长 = layer 数，宽 = seq position，深 = kv_dim 大小。这个盒子就是 KV cache。等会儿在 forward() 里你会看到代码怎么往这个盒子里写、怎么从里面读。

关键认知：这块内存在 CPU 上是一个普通 malloc，在你将来的 TRT 版里会变成一块 cudaMalloc 出来的 device memory。布局可以一模一样。

Section 3 · 30 min · search: void forward

3. forward() 函数 —— 整个 transformer 的一次前向

这是最长最密集的一段，也是最有价值的一段。不要逐行追细节， 按下面的顺序扫描，确认每一步在做什么。

扫描顺序（每一步 1-2 分钟）：

  1. token embedding 查表 —— x = w->token_embedding_table[token] 风格的代码
  2. 进入 layer 循环 for (int l = 0; l < p->n_layers; l++)
  3. 第一个 RMSNorm（rmsnorm 调用）
  4. QKV 投影（三次 matmul），把 x 变成 q、k、v
  5. RoPE 旋转位置编码（对 q 和 k 做）
  6. 关键：把 k 和 v 写进 key_cache 和 value_cache 的当前位置—— 找到那行赋值，看它是怎么用 l（layer）和 pos（当前 token 位置）算偏移的
  7. 注意力计算：q 和 整段 cache 里的 K 做点积 → softmax → 乘 整段 cache 里的 V → 输出
  8. O 投影 + 残差
  9. 第二个 RMSNorm + FFN（SwiGLU：w1 和 w3 算 gate，w2 投影回去）+ 残差
  10. 循环结束，最后 RMSNorm + classifier 投影到 logits（vocab_size 维）

要重点理解的一点：第 6 步把当前 token 的 K/V 写进 cache 之后，第 7 步的 attention 是用当前 q和cache 里 0 到 pos 的所有 K/V做的。这就是为什么 cache 必须存——下一步生成下一个 token 时，前面这些 K/V 都还要用。

动作：在你刚才画的 KV cache 盒子上，标出"当前这一步在第 l 层、第 pos 列写入数据"，然后画一个箭头表示"attention 时读 0 到 pos 这一段"。

Section 4 · 15 min · search: void generate

4. generate() 函数 —— autoregressive 循环

这一段是你将来在 TRTQwen2::generate() 里要复制的"骨架"。

要看的：

  • 外层循环结构：while (pos < steps) 之类
  • 每次循环里依次做了什么：调 forward(token, pos) → 拿到 state.logits → 调 sample() 拿下一个 token → 打印或保存 → pos++
  • 对 prompt 部分的特殊处理：prompt token 不用采样，直接把 prompt 里下一个 token 喂进去；只有当 pos 超过 prompt 长度后，才真正开始用 sampler 选 token
  • EOS 终止：哪个 token id 触发跳出循环

关键认知：llama2.c 的 generate 里其实没有显式的 "prefill 阶段"——它是一个 token 一个 token 喂进去的，prompt 也走同样的 forward。但 TRT 版会显式拆成 prefill + decode 两个 context：prefill 一次性吃整段 prompt（更高效），decode 每次只吃一个 token。这是 llama2.c 和你将来代码的最大结构差异。

Section 5 · 15 min · search: int sample

5. sample() 函数 —— 从 logits 到下一个 token

采样器是个独立小模块，llama2.c 里实现得很干净。这一段是你将来 Sampler 类的"参考实现"。

要看的：

  • greedy（argmax）路径：当 temperature 为 0 时直接返回 logits 最大的位置
  • temperature scaling：logits[i] /= temperature 然后过 softmax 变成概率
  • top-p（nucleus sampling）：把概率从大到小排，累加到超过 p 为止，只在这部分里采样
  • 多项式采样：从概率分布里抽一个

关键认知：这一整段都是纯 CPU 操作，不依赖 GPU。你将来在 toolkit 里第一版 Sampler 也可以纯 CPU——把 logits 从 device 拷回 host 再采样。后期再写 CUDA kernel 上 GPU 是独立优化（一篇博客）。

这 3 段可以扫一眼跳过

Skim · 5 min total

1. matmul / rmsnorm / softmax 的实现细节

知道它们做什么就够了，不用追每一行。在你将来的 TRT 版里，这些算子全部封装在 ONNX 模型 / TRT engine 里，你不需要自己写。

Skim · 5 min total

2. BPE tokenizer（bpe_encode 等函数）

llama2.c 用的是 Llama 的 SentencePiece tokenizer，和你将来要支持的 Qwen2 GPT2-style BPE 完全是不同的算法。这段算法层面没必要深读——你的 tokenizer 会单独写一份。

Skim · 5 min total

3. 模型加载（mmap、文件读取）

llama2.c 的 .bin 格式是 Karpathy 自己定的，TRT 加载的是 .engine 二进制。两者完全无关。这部分代码看一眼知道"哦，是个内存映射读权重"就够了。

自测：读完之后回答这 8 个问题

不用打字回答，心里默答或在笔记本上写一行就行。能答对 6 题，备课就到位了。

  1. KV cache 的总大小怎么算？给出公式，单位是字节。
  2. 第 l 层、第 pos 个 token 的 K 在 key_cache 数组里的偏移是多少？
  3. 每一步 forward 里，哪些张量需要"读取 cache 里所有历史位置"？
  4. RoPE 是在 K/V 写入 cache 之前还是之后做的？为什么这个顺序很关键？
  5. 从 logits（一个 vocab_size 维向量）到下一个 token id，中间经过哪几步操作？
  6. generate 里 prompt 的处理和后续生成的处理，区别在什么地方？
  7. 如果你要把 llama2.c 的 forward 改成"一次吃 N 个 token"（也就是 prefill 模式），要改哪几处？
  8. EOS token 是在哪个环节判断的？如果模型一直不输出 EOS，循环会怎么停下来？

判分标准：第 1、2、3、6 题答不上来 → 必读 5 段没看到位，回头再扫；第 4、5、7、8 题答不上来 → 没关系，工程性的细节，写代码时自然会清楚。

读完之后，把 llama2.c 的概念翻译到你的 TRT 版

这是这次精读的真正用途。下面这张表请收好，等你开始写 PR2 的 KVCacheBuffer 时再回来看一次。

llama2.c 里的东西 在你 TRT 版里对应什么
TransformerWeights struct 消失——所有权重打包在 .engine 文件里，你不直接持有
RunState 里的 x、q、k、v、logits 等中间张量 消失——TRT engine 内部分配，你不需要管
RunState 里的 key_cache + value_cache 变成你的 KVCacheBuffer，必须自己管理（device memory，跨调用持久化）
matmul、rmsnorm、RoPE、SwiGLU 这些算子 消失——TRT engine 替你跑
forward() 函数的算子序列 消失——一次 enqueueV3 替代
forward(token, pos) 的循环调用 变成你的 decode loop：每次调 decode_context->enqueueV3，配合 KV cache 的 slot 切换
prompt 阶段的逐 token forward 变成 prefill_context：一次性把整段 prompt 喂进去（高效得多）
sample() 函数 变成你的 Sampler 类，第一版直接照抄 llama2.c 的算法（CPU 实现）
BPE tokenizer 消失——你写自己的 GPT2-style BPE，从 vocab.json + merges.txt 加载
EOS 判断 + max_steps 截断 原样保留，写在 decode loop 里

看清楚了吗—— llama2.c 的 700 行里，只有"加粗"那 4 行是你要在 toolkit 里重新实现的。 这 4 行就是你的 PR2 + PR3 的全部业务代码。其他都被 TRT 替换掉，或者由 tokenizer 子模块独立处理。
