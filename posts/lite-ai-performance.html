lite.ai.performance：TensorRT 推理管线优化

2026.05.07 lite.ai TensorRT CUDA 性能优化

昨天的 YOLO 单次推理实验验证了一个方向：如果只优化 TensorRT engine， 很快会被 CPU 预处理、CPU 后处理、H2D/D2H 拷贝和同步点挡住。今天做的是把这套思路落到 lite.ai.toolkit 里，给 TensorRT YOLO 管线加一个真正面向延迟的 performance path。

目标是让 TRTYoloV8 的端到端延迟接近参考实现 /workspace/yolo_fast/infer_gpu.cu 的 1ms 级别。最终 FP16 engine 下， 新 GPU 管线 E2E 为 891us，GPU 耗时约 872us； 相比原始 CPU 前后处理管线的 44,978us，约 50 倍加速。

原管线的问题

原来的 TRTYoloV8::detect() 不是慢在某一个单点，而是整条链路都在付额外成本。 图像先在 CPU 上做 cvtColor、resize、convertTo、 HWC 到 CHW，再把 FP32 tensor 拷到 GPU。TensorRT 推理之后，又把完整输出 tensor 拷回 CPU， 在 CPU 上遍历 8400 个 anchor，生成候选框并做 NMS。

这条路径还有两个隐藏问题：一是 enqueueV3 前后存在多余同步； 二是即使 engine 是 FP16，输入侧也没有完整的 FP16 IO 支持。也就是说，模型算得再快， 也会被 CPU 计算、内存拷贝和同步点稀释掉。

这次优化的核心不是再换一个更快的 engine，而是把 CPU 夹在推理链路中间的工作尽量搬走。

新的数据流

新增的 detectGPU() 直接围绕设备端数据组织：原始图像以 uint8 形式进入 GPU， GPU kernel 完成前处理，TensorRT 直接消费 FP16 或 FP32 输入 tensor，候选框生成和 NMS 也尽量留在 GPU 端。 

d_raw(uint8, GPU)
  -> GPU preprocess kernel
     resize + BGR->RGB + normalize + HWC->CHW + FP16/FP32
  -> trt_context->enqueueV3(stream)
  -> GPU bbox generation
     8400 anchors -> device bbox buffer
  -> GPU NMS
  -> D2H copy final boxes

这样改完之后，端到端路径里最重的 CPU 操作被移除：不再在 CPU 上构造输入 tensor， 不再把完整 raw output 拉回 CPU 遍历，也不再在 CPU 上执行 NMS。CPU 主要负责启动 kernel、 管理 buffer、同步和取最终结果。

GPU 前处理

前处理被合并成一个 CUDA kernel，单次 launch 完成 letterbox resize、BGR 到 RGB、归一化、 HWC 到 CHW 和类型转换。这里同时提供 FP16 和 FP32 两种输出变体：FP16 engine 直接写 half* 输入，FP32 engine 写 float* 输入。 

// FP16: 写入 half* dst
preprocess_kernel<<<grid, block, 0, stream>>>(...)

// FP32: 写入 float* dst
preprocess_kernel_fp32<<<grid, block, 0, stream>>>(...)

这一步的意义和昨天 FP16 IO 的实验对应：如果 engine 输入 binding 本来就是 FP16， 就不要在 CPU 上生成 FP32 tensor 再传过去。直接在 GPU 上生成 FP16 CHW tensor， 能减少输入数据量，也能省掉额外转换。

FP16 输入支持

原实现里 FP16 engine 的输入支持不完整，甚至会出现按 FP16 大小分配 buffer， 但仍按 FP32 数据拷贝的情况。这会造成缓冲区溢出，也解释了旧管线里异常的检出数量。

新实现通过 TensorRT 的 getTensorDataType() 自动识别输入类型， 再按 sizeof(half) 或 sizeof(float) 分配输入 buffer。 对动态 shape，先 setInputShape，再绑定地址，并使用 getTensorShape() 获取实际维度。

GPU 后处理和 NMS

YOLOv8 后处理也搬到 GPU。候选框生成 kernel 让每个线程处理一个 anchor， 对 80 个类别分数取最大值作为置信度，符合阈值的候选框直接写入设备端数组。 后续 GPU NMS 直接消费这块设备内存，中间不需要把完整输出 tensor 拷回 CPU。

NMS 前会把内部的 BoundingBoxType<float, float> 打包成更适合 kernel 处理的 5-float 数组，然后在 GPU 上做逐类别 NMS。最终只把少量保留框和索引拷回 CPU。 这和昨天端到端 NMS 实验的结论一致：真正贵的不是几个最终框，而是把大输出搬回 CPU 再处理。

性能结果

测试环境是 RTX 3080 Ti，模型为 YOLOv8n，输入图像 810 x 1080， 置信度阈值 0.25，IOU 阈值 0.45，topk 为 100。

Engine Pipeline E2E 延迟 GPU 耗时 检出数
FP16 旧：CPU 前后处理 44,978us — 2750*
FP16 新：GPU 前后处理 891us 872us 5
FP32 旧：CPU 前后处理 44,745us — 2750*
FP32 新：GPU 前后处理 1,505us 1,462us 5

星号对应旧管线的 FP16 数据类型 bug：输出数量异常不代表真实检测质量， 而是 buffer 和 dtype 不匹配带来的错误结果。修复 dtype 后，新管线的检出数恢复正常。

延迟分解

FP16 新管线的主要耗时分布如下：

阶段 耗时
GPU 预处理 kernel ~10us
TRT enqueueV3 ~800us
GPU 后处理 kernel ~20us
GPU NMS ~40us
GPU 总计 ~872us
CPU 开销：memcpy、sync 等 ~19us
E2E 总计 ~891us

这个结果已经和参考实现的 1ms 级别接近。继续压均值延迟时，重点不再是“把 Python 换成 C++” 这种大块收益，而是减少同步点、合并后处理和 NMS、以及确认输入输出拷贝是否还能被流水线隐藏。

改动范围

这次改动主要集中在三层：TensorRT handler 层补齐 dtype 和动态 shape 支持； CUDA kernel 层新增 GPU 前处理、YOLO 后处理和 GPU NMS；CV API 层给 YOLOv8/YOLOv5 增加 detectGPU() 入口。CMake 也把 CUDA 架构调整为 sm_86，以适配 RTX 3080 Ti。

文件数量看起来不少，但边界是清楚的：原有 detect() 路径保留， 新的 performance path 通过 detectGPU() 进入，避免把低延迟实现和兼容性路径混在一起。

已知限制

  1. YOLOv5 的 detectGPU() 代码路径已经补上，但当前环境没有 YOLOv5 engine，尚未实测。
  2. YOLOv11 还没有适配 GPU 检测路径，仍然走 CPU 检测逻辑。
  3. 当前 detectGPU() 仍有同步点，后续可以考虑把后处理和 NMS 合并，或用 CUDA Graph 收敛调度开销。
  4. 这次测试聚焦单图低延迟，批量吞吐和多 stream 场景还需要单独 benchmark。

总结

这次优化最重要的判断是：lite.ai.toolkit 的 TensorRT YOLO 路径，瓶颈不只是 engine 推理本身。 旧管线把太多工作留在 CPU，并且反复跨越 CPU/GPU 边界。新的 performance path 把前处理、候选框生成和 NMS 放到 GPU 上，同时补齐 FP16 输入支持，把端到端延迟压到 891us。

这也验证了昨天 YOLO 单次推理实验里的结论：当模型已经进入 TensorRT FP16 的 1ms 区间后， 真正决定 E2E 的往往是 pipeline，而不是单独的 GPU compute。
