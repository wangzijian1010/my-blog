YOLOv8n 单次推理延迟优化实验

2026.05.06 YOLO TensorRT CUDA 推理优化

这次实验的目标很单纯：把 YOLOv8n 在 batch=1、imgsz=640 下的单次推理延迟尽可能压低。 测试环境是 RTX 3080 Ti、TensorRT 10.6、CUDA 13.0，验证集使用 COCO128。

最终结果是：PyTorch FP32 pipeline 的端到端延迟为 8.9ms；经过 TensorRT FP16、端到端 NMS、 FP16 输入、GPU 预处理和 C++ pipeline 之后，E2E 降到 1.17ms，约 7.6 倍加速。

先看完整结果

Pipeline mAP50 mAP50-95 Pre Infer Post E2E
PyTorch FP32 .pt 60.4 45.1 1.0ms 6.8ms 1.1ms 8.9ms
TRT FP32 engine 60.4 45.1 0.9ms 1.8ms 1.2ms 3.9ms
TRT FP16 engine 60.4 44.9 0.9ms 1.2ms 1.1ms 3.2ms
TRT INT8 engine 59.7 44.3 0.9ms 1.1ms 1.1ms 3.1ms
TRT FP16 NMS 59.2 44.0 0.9ms 2.0ms 0.2ms 3.1ms
C++ GPU Pipeline 59.2 44.0 0.01ms 1.02ms 0ms 1.17ms

前三步说明了一件事：只做 engine 精度量化，很快就会摸到上限。FP32 到 FP16 的收益很明显， 但 FP16 到 INT8 只把 infer 从 1.2ms 压到 1.1ms。此时 Pre + Post 仍然接近 2ms， 已经比模型本身更像瓶颈。

EfficientNMS 的收益和误区

我用 yolo export nms=True 导出端到端 ONNX，再通过 TensorRT build 成 engine。 ONNX 输入是 [1,3,640,640]，输出变成 [1,300,6]。 相比 raw 输出 [1,84,8400]，D2H 数据量从约 705K floats 变成 1.8K floats， 大约缩小 392 倍。

指标 FP16 raw FP16 NMS 变化
GPU Compute 0.80ms 0.79ms 基本不变
D2H 0.23ms 0.005ms 下降约 98%
Mean Latency 1.46ms 1.21ms 下降约 17%
Throughput 1131 QPS 817 QPS 下降

裸 engine 看起来不错：D2H 几乎没了，平均 latency 也下降。但放回 Ultralytics pipeline 后， 结果没有想象中漂亮。Postprocess 从 1.1ms 降到 0.2ms，确实达成目标； 但 NMS 被计入 inference，infer 从 1.2ms 增加到 2.0ms，最后 E2E 只从 3.2ms 到 3.1ms。

NMS 的主要收益不是“计算免费”，而是输出变小，CPU 后处理和 D2H 传输变便宜。

这里还有一个精度坑：第一次导出时使用了默认 conf=0.25，低分框被提前过滤， mAP50 从 60.4 掉到 47.9。验证时应该用 conf=0.001 导出。 修复后 mAP50 为 59.2，mAP50-95 为 44.0。Precision 上升、Recall 下降， 主要来自 ONNX NonMaxSuppression 和 TorchNMS 的实现差异。

FP16 输入是便宜收益

接下来把输入 binding 从 FP32 改成 FP16。640 x 640 x 3 的输入从 4.9MB 变成 2.45MB， H2D 理论上应该接近减半，实测也基本符合预期。

Metric FP32 IO FP16 IO 变化
H2D Latency 0.405ms 0.208ms 下降 49%
GPU Compute 0.788ms 0.795ms 基本不变
Mean Latency 1.184ms 1.013ms 下降 14%
Throughput 834 QPS 970 QPS 上升 16%

这一步几乎没有副作用。图像原始值是 0 到 255，normalize 到 0 到 1 后仍然在 FP16 可以表达的范围内。 对单次推理来说，它稳定省下约 0.2ms，让 trtexec 裸延迟第一次接近 1ms。

真正拉开差距的是 C++ GPU Pipeline

Python/Ultralytics pipeline 里，CPU 预处理、Python 调度和后处理已经成为主要成本。 所以后面改成 C++ TensorRT API，并把预处理放到 CUDA kernel 上做。 

CPU: stb_image(RGB) -> H2D raw uint8
GPU: resize + HWC->CHW + normalize -> TRT enqueueV3 -> D2H detections

预处理 kernel 每个 thread 处理一个输出像素，完成 bilinear resize、HWC uint8 到 CHW FP16、 normalize 到 0 到 1。输入直接用 RGB，省掉 BGR/RGB 转换。非默认 stream 也避免了一部分同步开销。

指标 延迟 说明
GPU Preprocess 9.75us 相比 Python CPU 0.9ms，约 93 倍加速
TRT Inference 1.024ms 包含 GPU compute 和 NMS
Pre + Infer 1.041ms CUDA event 测量
E2E wall 1.168ms 包含 H2D、preprocess、infer、D2H

这一步把 Python TRT pipeline 的 3.1ms 压到 1.17ms，下降约 62%； 对比最初的 PyTorch FP32 pipeline，则是 8.9ms 到 1.17ms，下降约 87%。

没有收益的实验也有价值

CUDA Graph 和 SpinWait 在这次场景里都没有收益。FP16 NMS engine 上，baseline、 --useCudaGraph、--useSpinWait 的 mean latency 都在 1.18ms 到 1.19ms 附近。 原因是 Enqueue ≈ H2D + GPU + D2H，说明 TensorRT 10.x 已经把模型编译成高度融合的执行图， 单图固定 shape 推理里没有多少 per-kernel launch overhead 可以省。

onnxsim 也没有带来变化。节点数从 288 到 288，说明 Ultralytics 内置的 onnxslim 已经完成了能做的 constant folding 和冗余节点消除。

当前瓶颈

C++ GPU pipeline 的 1.17ms 现在大致分成这几块：

GPU Compute (TRT + NMS) 0.80ms  68%
H2D raw uint8          0.21ms  18%
CPU overhead           0.15ms  13%
GPU Preprocess         0.01ms   1%
D2H detections         0.01ms   1%

也就是说，GPU compute 已经是主导项，接近这个模型和输入尺寸下的物理下限。 继续优化 Python、后处理、图简化，收益都不会太大。真正还可能有收益的方向， 要么降低输入分辨率，要么换模型结构，要么做 Ampere 2:4 structured sparsity。 如果目标是稳定性而不是均值延迟，GPU clock lock 和 P99 profiling 会更重要。

这次实验的结论

  1. TensorRT FP16 是第一优先级优化：延迟从 8.9ms 到 3.2ms，精度基本不变。
  2. INT8 对 YOLOv8n 单 batch 收益有限：infer 只从 1.2ms 到 1.1ms，还牺牲一点 mAP。
  3. 端到端 NMS 主要减少 D2H 和 CPU postprocess，但验证级 conf=0.001 下不会带来明显 E2E 净收益。
  4. FP16 IO 是低风险收益：H2D 下降约 49%，单次推理省约 0.2ms。
  5. C++ GPU pipeline 才是关键：GPU 预处理 + TRT API 消除 Python overhead 后，E2E 到 1.17ms。
  6. 当前瓶颈已经回到 GPU compute。后续优化应优先考虑分辨率、模型结构、稀疏化和流水线重叠。
