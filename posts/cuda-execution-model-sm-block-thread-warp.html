CUDA 编程模型入门：执行模型、内存层级、转置和 Reduce

2026.05.18 CUDA GPU SM Warp 性能优化

CUDA 编程模型的核心，就是把大量线程分组，然后让 GPU 的多个 SM 去并行执行这些线程组。 如果先不用术语，可以把它理解成一座工厂：GPU 是整座工厂，SM 是工厂里的车间， Block 是分配给车间的一批任务，Thread 是真正干活的工人，Warp 是 32 个工人组成的固定小队。 

一张 GPU = 一个工厂
SM       = 工厂里的车间
Block    = 分配给车间的一批任务
Thread   = 干活的工人
Warp     = 32 个工人组成的固定小队

你现在先记住这句话：CUDA 不是把一个大任务丢给一个“大核心”，而是把任务拆成很多 block， 再把这些 block 调度到很多 SM 上执行；每个 SM 内部又把线程按 warp 来执行。 后面所有概念都围绕这句话展开。

SM 是什么

SM 全称是 Streaming Multiprocessor，可以理解为 GPU 里真正干活的计算单元。 CPU 通常说 8 核、16 核；GPU 的结构不是一个超大的核心，而是由几十个甚至上百个 SM 组成。 每个 SM 里面有 CUDA Core、寄存器、共享内存、调度器等硬件资源。

所以你写 CUDA kernel 的时候，本质上是让很多线程被组织成 block， 再让 GPU 把这些 block 分配到不同 SM 上执行。你不直接调度 CUDA Core， 你真正需要重点理解的是 SM，因为 block 是被分配到 SM 上的。

3080 Ti 有多少个 SM

RTX 3080 Ti 使用 GA102 芯片，一共有 80 个 SM。 每个 SM 里有 128 个 FP32 CUDA Core，所以总 FP32 CUDA Core 数量是： 

80 × 128 = 10240 个 CUDA Core

但这里要注意：CUDA Core 不是你写代码时直接调度的单位。 你启动 kernel 时配置的是 grid 和 block，硬件调度时以 block 为单位分配到 SM。 理解了 SM，才算真正开始理解 CUDA 的执行模型。

Grid、Block、Thread 是什么关系

CUDA 里启动一个 kernel，一般长这样： 

kernel<<<gridDim, blockDim>>>(...);

比如： 

myKernel<<<4, 256>>>(...);

这句话的意思是：启动 4 个 block，每个 block 有 256 个 thread， 所以总线程数是： 

4 × 256 = 1024 个 thread

结构可以画成这样： 

Grid
 ├── Block 0
 │    ├── Thread 0
 │    ├── Thread 1
 │    ├── ...
 │    └── Thread 255
 │
 ├── Block 1
 │    ├── Thread 0
 │    ├── Thread 1
 │    ├── ...
 │
 ├── Block 2
 │
 └── Block 3

所以 CUDA 的三级结构是： 

Grid  →  Block  →  Thread
整个任务 → 任务小组 → 具体线程

一个重要细节是：一个 block 会被放到一个 SM 上执行，不会被拆到多个 SM 上。 这也是为什么同一个 block 内的线程可以使用 shared memory，也可以用 __syncthreads() 做同步；不同 block 之间不能直接同步，也不能直接共享 block 内的 shared memory。

CUDA 内置变量：blockIdx、threadIdx、blockDim、gridDim

这些变量是 CUDA 自动给你的。你不用自己定义，kernel 里可以直接用。

变量 含义 myKernel<<<4, 256>>> 里的值
threadIdx.x 当前线程在自己 block 里的编号 0 ~ 255
blockIdx.x 当前 block 在整个 grid 里的编号 0 ~ 3
blockDim.x 每个 block 有多少个线程 256
gridDim.x 整个 grid 有多少个 block 4

threadIdx 只是在当前 block 内部编号，不是全局编号。 所以 Block 0 里有 threadIdx.x = 0， Block 1 里也有 threadIdx.x = 0， Block 2 里也有 threadIdx.x = 0。 每个 block 内部都会从 0 重新编号。

怎么算一个线程的全局编号

这是 CUDA 最核心的公式之一： 

int idx = blockIdx.x * blockDim.x + threadIdx.x;

还是看这个启动配置： 

myKernel<<<4, 256>>>(...);

每个 block 里有 256 个线程，所以全局线程编号是： 

Block 0:
idx = 0 * 256 + threadIdx.x
idx = 0 ~ 255

Block 1:
idx = 1 * 256 + threadIdx.x
idx = 256 ~ 511

Block 2:
idx = 2 * 256 + threadIdx.x
idx = 512 ~ 767

Block 3:
idx = 3 * 256 + threadIdx.x
idx = 768 ~ 1023

所以这行代码的作用，就是把 block 内部编号转换成整个 grid 里的全局线程编号。 以后你写 CUDA kernel，十有八九都会看到它。

Block 是怎么分配到 SM 上的

当你启动： 

myKernel<<<100, 256>>>(...);

表示你创建了 100 个 block，每个 block 256 个 thread。 这些 block 会被 GPU 的硬件调度器分配到 SM 上执行。

如果你的 GPU 是 RTX 3080 Ti，有 80 个 SM，那么大概可以想成： 

SM0  分到一些 block
SM1  分到一些 block
SM2  分到一些 block
...
SM79 分到一些 block

但你不能手动指定某个 block 去某个 SM。CUDA 的调度规则可以粗略理解为： 哪个 SM 有空，哪个 SM 就拿下一个 block 执行。 

80 个窗口办理业务
100 个客户排队
哪个窗口空了，下一个客户就过去

窗口 = SM
客户 = Block

如果 block 数很多，SM 执行完当前驻留的一批 block 之后，会继续拿下一批 block。 这就是 GPU 能同时处理大量任务的基础。

一个 SM 能同时跑多少个 block

这个问题不能只看 GPU 型号，还要看你的 kernel 用了多少资源。 一个 SM 上可以同时驻留多个 block，这叫 resident blocks，也可以理解成同一个 SM 上同时挂着多个 block。

但一个 SM 能同时放几个 block，受这些因素限制：

  1. 每个 block 有多少线程。
  2. 每个线程用了多少寄存器。
  3. 每个 block 用了多少 shared memory。
  4. GPU 架构本身的硬件上限。

以 3080 Ti / Ampere 架构为例，一个 SM 的常见上限是： 

最多 1536 个 resident threads
最多 48 个 resident warps
最多 16 个 resident blocks

这些只是理论上限。你的 kernel 如果用了很多 shared memory 或寄存器，实际能驻留的 block 数会更少。

例子 1：每个 block 256 个线程

myKernel<<<..., 256>>>(...);

一个 block 有 256 个线程，也就是 8 个 warp。 如果只看线程数，一个 SM 最多 1536 个线程： 

1536 / 256 = 6 个 block

所以理想情况下，一个 SM 可以同时放 6 个 256-thread block。

例子 2：每个 block 1024 个线程

myKernel<<<..., 1024>>>(...);

一个 block 有 1024 个线程，也就是 32 个 warp。 只看线程数： 

1536 / 1024 = 1 个 block

虽然还剩 512 个线程容量，但不够再放一个 1024-thread block。 所以一个 SM 同时只能放 1 个这种 block。这也是为什么 block size 不是越大越好。

例子 3：每个 block 128 个线程

myKernel<<<..., 128>>>(...);

一个 block 有 128 个线程，也就是 4 个 warp。只看线程数： 

1536 / 128 = 12 个 block

所以一个 SM 理论上可以同时放 12 个 128-thread block。 但如果 kernel 使用了很多寄存器或 shared memory，实际数量可能低于 12。

Warp 是什么

Warp 是 CUDA 真正执行线程的基本单位。一个 warp 固定是 32 个线程。 你写代码的时候，感觉每个 thread 是独立执行的；但硬件执行的时候， 不是一个 thread 一个 thread 执行，而是 32 个 thread 组成一个 warp， 一起执行同一条指令。

这种执行模型叫 SIMT，Single Instruction, Multiple Threads。 可以理解成：一条指令，多个线程一起干。

比如这个最常见的向量加法 kernel： 

__global__ void add(float* a, float* b, float* c) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    c[idx] = a[idx] + b[idx];
}

假设一个 warp 里的 32 个线程编号是 thread 0 到 thread 31。 它们会一起执行： 

c[idx] = a[idx] + b[idx];

只是每个线程的 idx 不同： 

thread 0  处理 c[0]  = a[0]  + b[0]
thread 1  处理 c[1]  = a[1]  + b[1]
thread 2  处理 c[2]  = a[2]  + b[2]
...
thread 31 处理 c[31] = a[31] + b[31]

所以看起来是 32 个线程各干各的，实际上是硬件把它们作为一组来执行。

为什么 Warp 是理解性能问题的起点

GPU 性能问题很多都和 warp 有关。最重要的三个概念是：

  1. Warp divergence：分支发散。
  2. Memory coalescing：内存合并访问。
  3. Occupancy：SM 上有多少 warp 能同时驻留。

分支发散：warp divergence

看这个代码： 

if (threadIdx.x % 2 == 0) {
    // 偶数线程做 A
} else {
    // 奇数线程做 B
}

一个 warp 里有 32 个线程，其中 16 个线程走 if，16 个线程走 else。 问题是：一个 warp 里的线程应该一起执行同一条指令。 但现在一半要执行 A，一半要执行 B。

GPU 只能先让偶数线程执行 A，奇数线程暂时屏蔽； 再让奇数线程执行 B，偶数线程暂时屏蔽。 原来一次能干完的事情，现在变成两次，这就叫 warp divergence。

同一个 warp 里的线程走了不同分支，执行会被拆开，性能就会下降。

内存合并访问：memory coalescing

GPU 访问 global memory 的时候，很喜欢相邻线程访问相邻内存。 比如： 

c[idx] = a[idx] + b[idx];

一个 warp 里的线程访问： 

a[0], a[1], a[2], ..., a[31]

这些地址是连续的。硬件可以把这些访问合并成更少的内存事务， 这叫 coalesced memory access，也就是内存合并访问，性能很好。

但如果你这样访问： 

c[idx] = a[idx * 32] + b[idx * 32];

一个 warp 里的线程访问： 

a[0], a[32], a[64], a[96], ...

地址跳得很远，GPU 就很难合并访问，global memory 访问效率会下降。 所以你以后优化 CUDA kernel，经常会听到一句话：

让相邻线程访问相邻内存。

这句话的根源就是 warp。

Occupancy：占用率

Occupancy 可以粗略理解为：一个 SM 上同时挂了多少个 warp。 如果一个 SM 上只有很少的 warp，当某个 warp 等 global memory 的时候，SM 可能就没活干了。

如果一个 SM 上有很多 warp，那么： 

warp A 等内存
SM 切去执行 warp B

warp B 等内存
SM 切去执行 warp C

warp C 等内存
SM 再切去执行别的 warp

这样可以把等待时间隐藏掉。GPU 不是靠“单个线程很快”取胜， 而是靠大量 warp 切换来隐藏延迟。

但也要注意，occupancy 不是越高越好。它是重要指标，但不是唯一目标。 如果为了提高 occupancy 牺牲了数据复用、增加了访存或让寄存器溢出，性能也可能变差。 对入门阶段来说，先理解“足够多的 warp 可以隐藏延迟”就够了。

Block 和 Warp 的关系

一个 block 里面的线程，会被拆成多个 warp。公式是： 

warp 数 = ceil(blockDim.x / 32)

常见 block size 对应的 warp 数：

blockDim.x Warp 数
128 128 / 32 = 4
256 256 / 32 = 8
512 512 / 32 = 16
1024 1024 / 32 = 32

如果 blockDim.x 不是 32 的倍数，比如 100： 

ceil(100 / 32) = 4 个 warp

但是最后一个 warp 只有 4 个有效线程，剩下 28 个线程位置浪费。 所以 CUDA 里 block size 通常会设成 32 的倍数，比如 128、256、512。

完整例子：向量加法

假设你有两个数组： 

a = [1, 2, 3, 4, ...]
b = [10, 20, 30, 40, ...]

你想算： 

c[i] = a[i] + b[i]

CUDA kernel 可以这样写： 

__global__ void vectorAdd(float* a, float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

启动代码： 

int blockSize = 256;
int gridSize = (n + blockSize - 1) / blockSize;

vectorAdd<<<gridSize, blockSize>>>(a, b, c, n);

这里 blockSize = 256，表示每个 block 256 个线程。 gridSize 的计算是为了让总线程数覆盖 n 个元素： 

int gridSize = (n + blockSize - 1) / blockSize;

比如： 

n = 1000
blockSize = 256
gridSize = (1000 + 256 - 1) / 256 = 4

总线程数是： 

4 × 256 = 1024

但我们只需要处理 1000 个元素，所以最后 24 个线程会超出范围。 因此 kernel 里需要： 

if (idx < n)

这句判断是防止越界访问。几乎所有处理数组的 CUDA kernel 都会有类似保护。

CUDA 执行流程应该怎么理解

当你写： 

vectorAdd<<<gridSize, blockSize>>>(a, b, c, n);

实际发生的是：

  1. CPU 发起 kernel 启动请求。
  2. GPU 创建一个 grid。
  3. Grid 里包含很多 block。
  4. GPU 把 block 分配给各个 SM。
  5. 每个 SM 接收一个或多个 block。
  6. Block 内部线程被拆成 warp。
  7. Warp 以 32 个线程为一组执行指令。
  8. 每个 thread 根据自己的 idx 处理对应数据。

可以画成： 

GPU
├── SM 0
│   ├── Block 0
│   │   ├── Warp 0: Thread 0  ~ 31
│   │   ├── Warp 1: Thread 32 ~ 63
│   │   └── ...
│   └── Block 80
│
├── SM 1
│   ├── Block 1
│   └── Block 81
│
├── SM 2
│   ├── Block 2
│   └── Block 82
│
...
└── SM 79
    ├── Block 79
    └── Block 159

如果 block 数很多，SM 执行完一批 block 后，会继续拿下一批 block。 所以 block 数通常要足够多，否则你可能根本喂不满所有 SM。

面试里怎么说

如果面试官问 CUDA 的执行模型，可以这样回答：

CUDA 的执行模型是 Grid、Block、Thread 三级结构。一个 kernel 启动后会生成一个 Grid， Grid 由多个 Block 组成，每个 Block 又包含多个 Thread。Block 是调度到 SM 上的基本单位， GPU 会把不同的 Block 分配到不同 SM 上执行。一个 SM 可以同时驻留多个 Block， 但数量受线程数、寄存器、shared memory 和硬件上限影响。Thread 在硬件上不是单独调度执行， 而是以 Warp 为单位执行，一个 Warp 固定 32 个线程，采用 SIMT 模型。很多 CUDA 性能问题， 比如分支发散、内存合并访问、occupancy，本质上都要从 Warp 的执行方式来理解。

这个回答已经够专业了。再往深处追，就是具体架构、寄存器压力、shared memory、memory hierarchy、 Nsight profile 和具体 kernel 优化策略。

最应该记住的几个公式

全局线程 id

int idx = blockIdx.x * blockDim.x + threadIdx.x;

gridSize 计算

int gridSize = (n + blockSize - 1) / blockSize;

warp 数 = ceil(blockDim.x / 32)

256 线程 block 对应多少 warp

256 / 32 = 8 个 warp

执行模型先小结一下

CUDA 的核心不是“我启动了很多线程”这么简单，而是：

你启动很多 block；GPU 把 block 分配给不同 SM；SM 把 block 里的线程拆成 warp； warp 以 32 个线程为单位执行；性能好不好，主要看你的 warp 有没有高效执行、 内存访问有没有连续、SM 有没有足够多的活干。

理解到这个程度，就可以继续往 CUDA 性能优化走了。下一步不是背更多术语， 而是拿一个真实 kernel，用 Nsight 看它的 occupancy、memory throughput 和 warp 执行效率。

补充：CUDA 内存层级应该怎么学

学 CUDA 内存层级，不要一上来死背表。正确的学习方式是先建立一个空间感： 离计算单元越近，速度越快、容量越小、作用范围越小；离计算单元越远， 速度越慢、容量越大、作用范围越大。 

Register
    ↓
Shared Memory / L1 Cache
    ↓
L2 Cache
    ↓
Global Memory

越上面，越像线程和 SM 手边的小工具：快，但小，不能随便共享。 越下面，越像整个 GPU 的大仓库：容量大，所有线程都能访问，但延迟高。

CUDA 优化的核心，就是尽量减少 Global Memory 访问，把频繁复用的数据搬到更快的内存， 比如 Register 和 Shared Memory。

Register：线程自己的小本子

Register 是最快的存储。它是单个 thread 私有的，别的线程不能访问。 比如： 

float x = input[idx];
float y = x * 2.0f + 1.0f;
output[idx] = y;

这里的 x、y、idx 很可能会被编译器放到 register 里。 可以把 register 理解成每个线程手里拿着的小纸条：拿起来最快，但纸条数量有限， 而且只能自己看。

Register 是单线程私有的最快存储，常用于保存局部变量。虽然访问最快，但每个线程用太多 register 会降低一个 SM 上能同时驻留的线程数，从而降低 occupancy。

Shared Memory：一个 block 的公共黑板

Shared Memory 是 CUDA 里非常重要的手写优化工具。它的作用范围是同一个 block 内的线程， 不同 block 之间不能共享。可以理解成一个 block 里面所有线程共用的小黑板。 

__shared__ float tile[256];

int tid = threadIdx.x;
int idx = blockIdx.x * blockDim.x + tid;

tile[tid] = input[idx];
__syncthreads();

float value = tile[tid];

__shared__ float tile[256]; 声明了一块 block 内共享的 shared memory。 __syncthreads(); 表示 block 内所有线程同步一下，确保大家都把数据写完了， 后面的线程再去读就不会读到未完成的数据。

Shared Memory 是 block 内线程共享的片上内存，速度远快于 global memory。 它常用于 tiling：把 global memory 中会被重复访问的数据提前加载到 shared memory， 减少慢速显存访问。

L1 Cache：SM 自己的自动缓存

L1 Cache 不需要像 shared memory 那样手动声明，它是硬件自动工作的。 比如： 

float x = input[idx];

这个数据从 global memory 读的时候，如果命中 L1，就会快很多。 L1 Cache 可以理解成每个 SM 自带的小缓存，和 shared memory 一样都离 SM 很近， 都属于片上资源。

但二者的控制方式不一样：Shared Memory 是程序员手动控制， L1 Cache 是硬件自动控制。

L1 Cache 是 SM 局部缓存，主要缓存 global memory 访问。它由硬件自动管理， 而 shared memory 是程序员显式管理。很多架构中 L1 和 shared memory 共享片上存储资源。

L2 Cache：整张 GPU 的共享缓存

L2 Cache 是全 GPU 共享的，所有 SM 都能通过 L2 访问数据。 可以理解成整个工厂的公共仓库缓冲区。 如果某个数据会被多个 SM 访问，它有机会留在 L2 里，下次访问就不用直接去 global memory。

L2 Cache 是所有 SM 共享的缓存层，用来减少对 global memory 的直接访问。 多个 SM 之间的数据访问，有可能通过 L2 获得复用。

Global Memory：显存，大但慢

Global Memory 就是 GPU 显存。RTX 3080 Ti 有 12GB 显存， 模型权重、输入图片、输出 tensor、大数组，基本都放在 global memory 里。 它容量最大，但延迟最高。 

float x = input[idx];
output[idx] = x;

这里的 input 和 output 大概率都在 global memory。 所以很多 CUDA kernel 看起来只是简单读写，真正的性能瓶颈却在显存访问。

Global Memory 是 GPU 上容量最大的内存，所有线程都可以访问，但延迟最高。 CUDA 优化通常围绕三件事展开：减少 global memory 访问次数、提高访问连续性、 提高带宽利用率。

用矩阵乘法理解内存层级

矩阵乘法是 CUDA 面试最常见的例子。假设： 

C = A × B

每个 C 元素都要用到 A 的一行和 B 的一列。 最普通的写法大概是： 

for (int k = 0; k < K; k++) {
    sum += A[row * K + k] * B[k * N + col];
}

问题在于：A 和 B 都在 global memory， 每次循环都从 global memory 读，global memory 又很慢。 如果很多线程反复读取相同区域的数据，却每次都回到显存拿，性能就会很差。

Tiling 的核心思路

Tiling 的想法是：不要每次都直接从 global memory 读。 先把一小块 A 和 B 搬到 shared memory， 然后 block 里的线程反复使用这一小块数据。 

Global Memory → Shared Memory → Register → 计算

换成大白话就是： 

慢速大仓库
    ↓
block 内小黑板
    ↓
线程自己的小纸条
    ↓
计算

Shared memory 不是魔法，它真正解决的是数据复用： 一批线程合作从 global memory 读一次，放到 shared memory， 然后大家反复用，避免每个线程都重复去 global memory 读很多次。

所以 shared memory 的主战场不是单纯 copy，而是矩阵乘法、卷积、reduce、scan、transpose 这类“同一批数据会被多个线程重复使用”的场景。

Coalesced Access 和内存层级的关系

Coalesced access 一般翻译成合并访问，或者连续内存访问。 它解决的是另一个问题：如果你不得不访问 global memory，怎么访问得更高效？

GPU 喜欢这种访问： 

thread 0  访问 input[0]
thread 1  访问 input[1]
thread 2  访问 input[2]
...
thread 31 访问 input[31]

也就是同一个 warp 内，相邻线程访问相邻地址。 这样硬件可以把 32 个线程的访问合并成少数几次内存事务。

不好的访问是： 

thread 0 访问 input[0]
thread 1 访问 input[1000]
thread 2 访问 input[2000]
thread 3 访问 input[3000]

地址跳得很远，global memory 访问效率就会差。 这也是前面讲 warp 时说“让相邻线程访问相邻内存”的原因。

Shared Memory 是减少 Global Memory 访问次数；Coalesced Access 是让必须发生的 Global Memory 访问尽量高效。

Occupancy 和内存层级的关系

Occupancy 粗略理解就是一个 SM 上同时能挂多少个 warp。 为什么需要很多 warp？因为 global memory 很慢。 一个 warp 访问 global memory 时，可能要等几百个 cycle。

这时候 SM 不会原地等，它会切换去执行别的 warp： 

warp A 等 global memory
SM 执行 warp B

warp B 等 global memory
SM 执行 warp C

warp C 等 global memory
SM 执行 warp D

这叫隐藏内存延迟。Occupancy 的意义，就是让 SM 上有足够多的 warp， 当某些 warp 等待慢速内存时，SM 可以切换去执行其他 warp。

但 occupancy 也会被 register 和 shared memory 限制。 如果一个 kernel 每个线程用了很多 register，或者每个 block 用了很多 shared memory， 那一个 SM 上能同时驻留的 block 和 warp 就会变少。

Register 和 shared memory 都是 SM 上的有限资源。每个线程使用的 register 越多、 每个 block 使用的 shared memory 越多，一个 SM 能同时驻留的 block/warp 就越少， 可能降低 occupancy。因此 CUDA 优化不是简单地多用 shared memory， 而是要在数据复用和 occupancy 之间做平衡。

两个最小代码例子

普通 global memory 访问

__global__ void copyKernel(float* input, float* output, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        output[idx] = input[idx];
    }
}

这个 kernel 做的事情很简单：从 global memory 读 input[idx]， 再写回 global memory 的 output[idx]。 它计算太少，瓶颈大概率是 global memory bandwidth。 

input[idx] 从 global memory 读
    ↓
可能经过 L2
    ↓
可能经过 L1
    ↓
放到 register
    ↓
写回 output[idx] 到 global memory

所以它不是简单的“读一下写一下”，背后实际走的是整套内存层级。

Shared Memory 的代码形式

__global__ void sharedExample(float* input, float* output) {
    __shared__ float smem[256];

    int tid = threadIdx.x;
    int idx = blockIdx.x * blockDim.x + threadIdx.x;

    smem[tid] = input[idx];
    __syncthreads();

    output[idx] = smem[tid];
}

这段代码本身不一定比直接 copy 更快，因为它没有真正的数据复用。 但它展示了 shared memory 的基本使用方式： __shared__ float smem[256]; 声明 block 内共享数组； 每个线程从 global memory 读一个元素到 shared memory； __syncthreads(); 确保 block 内线程都写完； 最后再从 shared memory 读。

真正有用的是一个数据被多个线程反复使用的场景，比如矩阵乘法、卷积、reduce、scan 和 transpose。

面试里可能怎么问

CUDA 的内存层级有哪些

CUDA 中常见的内存层级包括 register、shared memory、L1 cache、L2 cache 和 global memory。 Register 是线程私有的，速度最快；shared memory 是 block 内线程共享的片上内存， 适合做数据复用；L1 cache 是 SM 级别缓存；L2 cache 是全 GPU 共享缓存； global memory 是显存，容量最大但延迟最高。优化时一般尽量减少 global memory 访问， 把可复用数据放到 register 或 shared memory，同时保证 global memory 访问连续合并。

Shared Memory 和 Global Memory 有什么区别

Global memory 是显存，所有线程都可以访问，容量大但延迟高。Shared memory 是 SM 上的片上内存， 只能被同一个 block 内的线程共享，容量小但速度快得多。Shared memory 通常用于缓存 global memory 中会被重复使用的数据，比如矩阵乘法里的 tile，从而减少对 global memory 的访问次数。

Register 用得越多越好吗

不是。Register 访问最快，但每个 SM 的 register 数量有限。如果每个线程使用太多 register， 会导致一个 SM 上能同时驻留的线程数或 warp 数减少，从而降低 occupancy。 Occupancy 降低后，GPU 隐藏 global memory 延迟的能力可能变差，所以 register 使用需要和 occupancy 做平衡。

为什么 Global Memory 慢还要用

因为 global memory 容量最大，模型权重、输入输出 tensor、大数组都必须放在 global memory 中。 优化不是完全不用 global memory，而是减少不必要的访问，并让访问模式尽量 coalesced， 提高带宽利用率。

Shared Memory 一定比 L1 Cache 好吗

不一定。Shared memory 的优势是程序员可以显式控制数据布局和复用，非常适合 tiling 这种确定的数据复用场景。 L1 cache 是硬件自动管理，使用简单但可控性较差。如果访问模式规则、复用明确，shared memory 往往更可控； 如果访问模式简单连续，L1 cache 可能已经足够。

看到一个 CUDA kernel，先问这 4 个问题

  1. 数据主要从哪里来？global memory 还是 shared memory？
  2. 同一个数据会不会被多个线程重复使用？
  3. Warp 内线程访问 global memory 是否连续？
  4. Register / shared memory 用量会不会影响 occupancy？

这 4 个问题，就是 CUDA 性能分析的入口。先问清楚它们，再谈具体优化手段， 你的分析就不会散。

向量加法到底在学什么

向量加法代码很简单： 

__global__ void vec_add(float* a, float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

它不是为了考你会不会加法，而是让你掌握 CUDA kernel 的最小骨架。 这里的全局索引公式前面已经讲过，再换一种角度复习一遍。

为什么这样算全局索引

int idx = blockIdx.x * blockDim.x + threadIdx.x;

因为 threadIdx.x 只是 block 内部编号。 比如： 

vec_add<<<4, 256>>>(...);

表示 4 个 block，每个 block 256 个线程。 每个 block 里的 threadIdx.x 都会从 0 开始： 

Block 0: threadIdx.x = 0 ~ 255
Block 1: threadIdx.x = 0 ~ 255
Block 2: threadIdx.x = 0 ~ 255
Block 3: threadIdx.x = 0 ~ 255

如果只用 int idx = threadIdx.x;，每个 block 都会重复处理 0 到 255。 所以必须加上 block 的偏移量： 

blockIdx.x * blockDim.x

完整映射就是： 

Block 0:
idx = 0 * 256 + threadIdx.x
idx = 0 ~ 255

Block 1:
idx = 1 * 256 + threadIdx.x
idx = 256 ~ 511

Block 2:
idx = 2 * 256 + threadIdx.x
idx = 512 ~ 767

Block 3:
idx = 3 * 256 + threadIdx.x
idx = 768 ~ 1023

所以这个公式的本质，就是把 block 内线程编号转换成整个 grid 里的全局线程编号。

为什么需要边界判断

这点前面也碰到过：grid size 通常是向上取整。 

int block = 256;
int grid = (n + block - 1) / block;

如果 n = 1000，block = 256： 

grid = (1000 + 255) / 256 = 4
实际启动线程数 = 4 * 256 = 1024

但是你只有 1000 个元素，最后 24 个线程的 idx 是 1000 到 1023。 如果没有： 

if (idx < n)

这些线程就会访问 a[1000]、a[1001] 这些越界地址。

因为 kernel launch 的线程数量通常按 block size 向上取整，不一定刚好等于 n， 所以需要用 if (idx < n) 防止多出来的线程访问越界。

为什么 block 常选 128、256、512

这也和前面讲过的 warp 直接相关。Warp size 是 32， 一个 block 里的线程会被拆成多个 warp： 

128 threads = 4 warps
256 threads = 8 warps
512 threads = 16 warps

它们都是 32 的整数倍。如果选 1000 threads，虽然没有超过单个 block 常见的 1024 线程上限， 但 1000 不是 32 的整数倍： 

1000 / 32 = 31.25

实际会变成 32 个 warp，最后一个 warp 只有 8 个有效线程，剩下 24 个 lane 浪费。 另外，block 太大也不一定好。比如 1024 线程一个 block， 可能导致一个 SM 同时驻留的 block 数变少，不利于调度和隐藏延迟。

所以工程里常见选择是 128、256、512。初学阶段默认写 int block = 256; 是一个很稳的选择。

矩阵转置到底在学什么

矩阵转置是 CUDA 入门优化的重头戏。它不是因为数学难，而是因为它完美暴露了 CUDA 内存访问的问题： 读和写很难同时做到连续访问。 

out[x][y] = in[y][x]

如果输入矩阵是： 

1 2 3
4 5 6
7 8 9

转置后是： 

1 4 7
2 5 8
3 6 9

C/C++ 里的二维数组按 row-major 存储，也就是： 

matrix[y][x]  →  matrix[y * N + x]

这点必须滚瓜烂熟。后面判断访问是否连续，看的就是同一个 warp 里 threadIdx.x = 0 ~ 31 时，一维地址怎么变化。

版本 1：Naive transpose

__global__ void transpose_naive(float* in, float* out, int N) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;
    int y = blockIdx.y * blockDim.y + threadIdx.y;

    if (x < N && y < N) {
        out[x * N + y] = in[y * N + x];
    }
}

这里假设矩阵按 row-major 存储，所以 in[y][x] 在一维数组里是 in[y * N + x]。

Naive 版本读是连续的

Coalesced access 前面已经讲过：同一个 warp 内相邻线程访问连续 global memory 地址， 硬件就能把多个请求合并成更少的 memory transaction。

对同一个 warp，如果 threadIdx.x 连续变化，读地址是： 

thread 0  读 in[y * N + x + 0]
thread 1  读 in[y * N + x + 1]
thread 2  读 in[y * N + x + 2]
...
thread 31 读 in[y * N + x + 31]

地址连续，所以读是 coalesced 的，很好。

Naive 版本写是不连续的

写入地址是： 

out[x * N + y]

对同一个 warp： 

thread 0  写 out[(x + 0) * N + y]
thread 1  写 out[(x + 1) * N + y]
thread 2  写 out[(x + 2) * N + y]
...
thread 31 写 out[(x + 31) * N + y]

地址之间相隔 N 个 float。如果 N 很大，地址跳得非常远。 这就是 strided access，也就是非连续访问。

Naive transpose 的问题不是计算复杂，而是 global memory 访问模式差。 它读是 coalesced 的，但写是 strided access，导致同一个 warp 的线程写入地址不连续， 无法高效合并访问。

版本 2：Shared memory transpose

Shared memory 前面已经讲过：它是同一个 block 内线程共享的片上内存， 比 global memory 快很多，常用于缓存和数据重排。 在矩阵转置里，shared memory 的作用不是“存更多数据”，而是做一次中转， 让 global memory 的读和写都变成连续访问。 

__global__ void transpose_shared(float* in, float* out, int N) {
    __shared__ float tile[32][32];

    int x = blockIdx.x * 32 + threadIdx.x;
    int y = blockIdx.y * 32 + threadIdx.y;

    tile[threadIdx.y][threadIdx.x] = in[y * N + x];
    __syncthreads();

    x = blockIdx.y * 32 + threadIdx.x;
    y = blockIdx.x * 32 + threadIdx.y;

    out[y * N + x] = tile[threadIdx.x][threadIdx.y];
}

第一阶段：连续读 global memory

tile[threadIdx.y][threadIdx.x] = in[y * N + x];

对一个 warp 来说，threadIdx.x 连续变化， 所以读取的是： 

in[y * N + x]
in[y * N + x + 1]
in[y * N + x + 2]
...

这是连续读，global memory 读取是 coalesced 的。

第二阶段：放到 shared memory 里

读进来的数据先放到 tile[threadIdx.y][threadIdx.x]， 也就是一个 32 x 32 的小块。可以理解成： 

从 global memory 搬一块 32x32 到 shared memory

这里的 __syncthreads(); 前面已经碰到过。 它表示 block 内所有线程在这里集合，等大家都完成 shared memory 写入后， 再继续往下执行。否则有些线程可能会读取到还没写好的 shared memory 数据。

第三阶段：连续写 global memory

x = blockIdx.y * 32 + threadIdx.x;
y = blockIdx.x * 32 + threadIdx.y;

out[y * N + x] = tile[threadIdx.x][threadIdx.y];

这一步把 block 坐标交换了。原来处理的是输入矩阵中的一块， 转置后要写到输出矩阵中对称的位置，所以 blockIdx.x 和 blockIdx.y 的角色发生了交换。

关键是写的时候，threadIdx.x 仍然让 x 连续变化。 因此 out[y * N + x] 的写入地址也是连续的。

Shared memory 版本把 naive 的“读连续、写不连续”，变成了“读 global memory 连续、 写 global memory 也连续”。中间靠 shared memory 做了一次转置缓冲。

但这个版本还有一个问题：虽然 global memory 访问变好了， shared memory 自己可能出问题。问题就出在： 

tile[threadIdx.x][threadIdx.y]

这相当于按列读取 shared memory，可能导致 bank conflict。 Bank conflict 是前面没有展开的新概念，下面单独讲。

Shared Memory Bank 是什么

Shared memory 不是一个完整的大桶，它被分成多个 bank。 常见情况下可以先按这个模型理解： 

32 个 bank
每个 bank 宽度 4 字节
float 正好 4 字节

所以对 float 数组来说，连续的 float 通常会依次落到 bank 0、bank 1、bank 2， 一直到 bank 31： 

tile[0][0]  → bank 0
tile[0][1]  → bank 1
tile[0][2]  → bank 2
...
tile[0][31] → bank 31

如果同一个 warp 的 32 个线程访问 32 个不同 bank，速度很好。 如果多个线程访问同一个 bank 的不同地址，就会发生 bank conflict， 这些访问会被拆开处理，shared memory 访问就会变慢。

Bank conflict 的核心：同一个 warp 里多个线程同时访问同一个 shared memory bank 的不同地址， 硬件无法一次并行完成，只能分批处理。

为什么 tile[32][32] 会有 bank conflict

二维数组在内存里是 row-major 存储： 

float tile[32][32];

内存布局是： 

tile[0][0], tile[0][1], ..., tile[0][31],
tile[1][0], tile[1][1], ..., tile[1][31],
tile[2][0], ...

地址下标可以理解成： 

address_index = row * 32 + col
bank = address_index % 32

按行访问没有冲突

访问 tile[threadIdx.y][threadIdx.x] 时， 假设 threadIdx.y 固定，threadIdx.x = 0 ~ 31： 

tile[row][0]  → index = row * 32 + 0  → bank 0
tile[row][1]  → index = row * 32 + 1  → bank 1
tile[row][2]  → index = row * 32 + 2  → bank 2
...
tile[row][31] → index = row * 32 + 31 → bank 31

32 个线程访问 32 个不同 bank，没有冲突。

按列访问会冲突

访问 tile[threadIdx.x][threadIdx.y] 时， 假设 threadIdx.y 固定，也就是列 col 固定， threadIdx.x = 0 ~ 31： 

tile[0][col]  → index = 0  * 32 + col
tile[1][col]  → index = 1  * 32 + col
tile[2][col]  → index = 2  * 32 + col
...
tile[31][col] → index = 31 * 32 + col

bank 计算是： 

bank = (row * 32 + col) % 32

因为 row * 32 % 32 = 0，所以： 

bank = col

也就是说： 

tile[0][col]  → bank col
tile[1][col]  → bank col
tile[2][col]  → bank col
...
tile[31][col] → bank col

32 个线程全部访问同一个 bank，这就是 32-way bank conflict。

为什么 tile[32][33] 能解决 bank conflict

padding 版本只改一行： 

__shared__ float tile[32][33];

第二维从 32 变成 33，地址计算就变成： 

address_index = row * 33 + col
bank = address_index % 32

如果按列访问： 

tile[0][col]  → index = 0  * 33 + col → bank = col
tile[1][col]  → index = 1  * 33 + col → bank = col + 1
tile[2][col]  → index = 2  * 33 + col → bank = col + 2
...
tile[31][col] → index = 31 * 33 + col → bank = col + 31

因为： 

33 % 32 = 1

所以每往下一行，bank 就错开 1 个。 于是原来全部落到同一个 bank 的列访问，被分散到了 32 个不同 bank。

Padding 的本质不是让 shared memory 变大一点，而是改变二维数组每一行的步长， 让按列访问时的 bank 映射错开，从而消除 32-way bank conflict。

三个转置版本应该怎么学

不要一上来追性能数字。更稳的顺序是先画图，再写代码，再测性能。

第一步：先画小矩阵

先画一个小矩阵，不要直接画 32 x 32： 

原矩阵 in:
a00 a01 a02 a03
a10 a11 a12 a13
a20 a21 a22 a23
a30 a31 a32 a33

转置后 out:
a00 a10 a20 a30
a01 a11 a21 a31
a02 a12 a22 a32
a03 a13 a23 a33

先理解： 

in[y][x] → out[x][y]
out[x * N + y] = in[y * N + x]

第二步：手敲 naive

写完 naive 后，只问自己两个问题：读是不是连续？写是不是连续？ 判断方法就是看同一个 warp 里 threadIdx.x = 0 ~ 31 时， 它们访问的一维地址是不是连续。

第三步：手敲 shared 版本

重点理解两个动作： 

global memory → shared memory
shared memory → global memory

也就是： 

tile[ty][tx] = in[y * N + x];
out[y * N + x] = tile[tx][ty];

Shared memory 在这里是中转站，用来改变访问顺序，让 global memory 的读写都尽量连续。

第四步：手敲 padded 版本

只改一行： 

__shared__ float tile[32][33];

然后用纸算 tile[0][col]、tile[1][col]、 tile[2][col] 到 tile[31][col] 分别落在哪个 bank。 这一段必须手算一遍，bank conflict 才会真的懂。

计时和有效带宽

计时是这部分的新实践点。不要用 std::chrono 直接包 kernel， 因为 kernel launch 是异步的。CPU 发起 kernel 后可能马上返回， 但 GPU 还没有真正执行完。

更合适的是用 CUDA event： 

cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start);
transpose_kernel<<<grid, block>>>(d_in, d_out, N);
cudaEventRecord(stop);

cudaEventSynchronize(stop);

float ms = 0;
cudaEventElapsedTime(&ms, start, stop);

cudaEventSynchronize(stop); 很关键。 它确保 stop event 之前的 GPU 工作已经完成，否则你测到的时间不可靠。

矩阵转置不是算力密集型，而是内存带宽密集型。 一次转置至少要读 N * N 个 float，再写 N * N 个 float。 所以总数据量是： 

N * N * sizeof(float) * 2

有效带宽可以这样算： 

double bytes = 2.0 * N * N * sizeof(float);
double gbps = bytes / (ms / 1000.0) / 1e9;

你不需要一开始就追求某个固定 GB/s 数字。 先看趋势：naive 慢，shared 快，padded 更合理。 如果结果不是这样，反而是学习机会。常见原因包括矩阵太小、计时包含了 cudaMemcpy、没有 warmup、没有重复运行取平均、block/grid 配置不对、 编译没开优化、边界判断有问题。

Nsight Compute 这一步在学什么

Nsight Compute 不要求你马上变成 profiling 专家。 这一步的目标只是建立一个感觉：代码慢，工具能告诉你慢在哪里。 先认识几个指标就够。

Memory Throughput

看 global memory 带宽用了多少。如果 naive 很低，shared/padded 明显更高， 通常说明 coalescing 和访问重排确实起作用了。

Shared Memory Bank Conflicts

看 shared memory 是否发生 bank conflict。 理论上 tile[32][32] 版本应该有明显 bank conflict， tile[32][33] 版本应该明显减少，甚至接近 0。

Achieved Occupancy

Occupancy 前面已经讲过。这里看的是实际活跃 warp 的比例。 它不是越高一定越好，但太低通常需要注意：可能是 block size、register 用量、 shared memory 用量限制了 SM 上能驻留的 warp 数。

Memory Coalescing

看 global memory 访问是否合并良好。Naive transpose 的写入通常比较差； shared/padded 版本的 global memory 读写应该更好。

这部分的 5 个自检题

3080 Ti 上一个 SM 最多能驻留多少 warp 和 block

Ampere / RTX 3080 Ti 上，每个 SM 最多 48 个 resident warps、1536 个 resident threads、 16 个 resident blocks。但这是硬件理论上限，实际 occupancy 还会受 block size、 每个线程使用的 register 数、每个 block 使用的 shared memory 限制。

为什么 warp size 是 32

Warp size 是 NVIDIA CUDA 硬件定义的执行粒度，一个 warp 固定 32 个线程。 它是硬件在吞吐、调度复杂度、内存合并访问粒度和控制逻辑之间做的架构选择。 对 CUDA 程序员来说，重点不是追问为什么一定是 32，而是理解硬件以 32 个线程为一组执行， 所以 block size 通常选择 32 的整数倍，并尽量避免同一个 warp 内的分支发散和非连续内存访问。

Shared memory 和 L1 cache 是什么关系

Shared memory 和 L1 cache 都是 SM 附近的片上存储资源。L1 cache 由硬件自动管理， 主要缓存 global memory 访问；shared memory 由程序员显式声明和管理， 用于 block 内线程共享数据。很多 NVIDIA 架构中，L1 cache 和 shared memory 会共享同一部分片上存储资源， 只是使用方式不同。

什么叫 coalesced access，非 coalesced 会损失多少性能

Coalesced access 指同一个 warp 内相邻线程访问连续的 global memory 地址。 这样硬件可以把多个线程的内存访问合并成更少的 memory transaction，提高带宽利用率。 非 coalesced access 会产生更多内存事务，导致实际带宽大幅下降。 具体损失多少取决于访问跨度、数据类型、缓存命中和架构， 但像矩阵转置这种典型 strided write，性能可能差好几倍甚至一个数量级。

Bank conflict 是怎么产生的，为什么 padding 能解决

Shared memory 被划分成 32 个 bank。对于 float 类型，连续元素通常落到连续 bank。 如果同一个 warp 中多个线程访问同一个 bank 的不同地址，就会发生 bank conflict，访问会被串行化。 tile[32][32] 按列访问时，相邻行地址间隔是 32 个 float， 对 32 个 bank 取模后都落在同一个 bank，造成 32-way conflict。 把第二维改成 33 后，相邻行地址间隔变成 33，33 对 32 取模等于 1， 所以每一行错开一个 bank，列访问时分散到不同 bank，从而消除冲突。

这部分最应该带走的东西

不要把今天理解成“我学了 vec_add 和 transpose”。 更准确的理解应该是：你学了 CUDA 性能优化的第一套基本功： 线程如何映射到数据；warp 如何访问 global memory；shared memory 如何做数据重排； bank conflict 为什么会拖慢 shared memory；如何用 cudaEvent 和 Nsight Compute 验证性能问题。

向量加法让我熟悉 CUDA kernel 的基本写法和线程索引；矩阵转置让我理解 CUDA 优化的核心不是数学操作本身， 而是内存访问模式。Naive transpose 的问题是写 global memory 不连续， shared memory 版本通过 tile 做中转，让读写都变成 coalesced； 但 tile[32][32] 按列访问会导致 shared memory bank conflict， 所以通过 tile[32][33] padding 让相邻行错开 bank，从而消除冲突。 最后用 cudaEvent 和 Nsight Compute 验证性能差异。

Day 2：Reduce 优化到底在学什么

Day 2 真正要学的不是“求和”，而是 CUDA 优化的完整套路。 Reduce 做的事情看起来很简单： 

sum = a[0] + a[1] + a[2] + ... + a[n - 1];

CPU 上一个 for 循环就能写完。但 GPU 上难点在于：GPU 擅长大量线程并行干活， 而 reduce 最终要把很多数合成一个数，天然有“汇聚”和“依赖”。 所以 Day 2 的核心是：

怎么把一个看起来串行的求和问题，改造成 GPU 适合的并行树形规约。

这一节会继续沿用前面已经讲过的概念：线程索引、shared memory、__syncthreads()、 warp divergence、bank conflict、occupancy、memory bound 和 cudaEvent。 新概念包括：tree-based reduction、first add during load、volatile、warp shuffle、 shuffle mask、grid-stride loop，以及多轮 partial sum 规约。

Reduce 本质到底是什么

假设有 8 个数： 

a0 a1 a2 a3 a4 a5 a6 a7

CPU 通常这样算： 

float sum = 0;
for (int i = 0; i < n; i++) {
    sum += a[i];
}

它的实际顺序更像： 

sum = ((((((a0 + a1) + a2) + a3) + a4) + a5) + a6) + a7

这是串行的，只有一个执行流在干活。GPU 更希望这样做： 

第 1 轮：
a0 + a1     a2 + a3     a4 + a5     a6 + a7

第 2 轮：
a0+a1+a2+a3             a4+a5+a6+a7

第 3 轮：
a0+a1+a2+a3+a4+a5+a6+a7

这叫 Tree-based Reduction，也就是树形规约。它的特点是： 

总计算量：O(N)
规约深度：O(log N)

比如有 1024 个数，规模变化是： 

1024 → 512 → 256 → 128 → ... → 1

大概只需要 log2(1024) = 10 轮。 GPU 上 reduce 的核心思想，就是每一轮让一半线程把另一半数据加到自己身上， 数据规模不断减半。

为什么 Reduce 是 CUDA 面试高频

Reduce 串起了前面学过的大部分基础优化点：

  1. 线程索引。
  2. Shared memory。
  3. __syncthreads()。
  4. Warp divergence。
  5. Bank conflict。
  6. Occupancy。
  7. Memory bound。
  8. Loop unrolling。
  9. Warp shuffle。
  10. Grid-stride loop。
  11. 多 block 结果如何再次规约。

面试官问 reduce，通常不是想知道你会不会加法，而是想知道： 你是否理解一个 CUDA kernel 从低效到高效的演进逻辑。

完整 Reduce 不是一个 kernel 就结束

这是 reduce 里很容易漏掉的工程细节。 假设有 n = 16M 个 float，你不可能只用一个 block 算完整个数组。 通常做法是： 

第一轮 kernel：
每个 block 负责一段数据
每个 block 算出一个 partial sum
输出到 output[blockIdx.x]

第二轮：
再对这些 partial sum 做 reduce

第三轮：
如果 partial sum 还很多，继续 reduce

数据流是： 

input[N] → partial_sum[numBlocks] → final_sum[1]

所以很多教学里的 reduce_v1 到 reduce_v6， 本质上通常只是在做 block 内部的一次局部规约，输出一个 block 级 partial sum。

单个 reduce kernel 通常只完成 block 级规约，得到每个 block 的 partial sum。 最终结果需要继续用第二个 kernel 递归规约这些 partial sum， 或者把 partial sum 拷回 CPU 做最后求和。工业实现里通常会多轮 kernel， 或者用 grid-stride loop 减少 partial sum 数量。

V1：Interleaved Addressing 在干什么

V1 是最朴素的 shared memory 树形规约： 

for (int s = 1; s < blockDim.x; s *= 2) {
    if (tid % (2 * s) == 0) {
        sdata[tid] += sdata[tid + s];
    }
    __syncthreads();
}

假设 blockDim = 8，一开始 shared memory 里是： 

sdata[0] sdata[1] sdata[2] sdata[3] sdata[4] sdata[5] sdata[6] sdata[7]

当 s = 1，条件是 tid % 2 == 0， 工作线程是 0, 2, 4, 6： 

sdata[0] += sdata[1]
sdata[2] += sdata[3]
sdata[4] += sdata[5]
sdata[6] += sdata[7]

当 s = 2，条件是 tid % 4 == 0， 工作线程是 0, 4： 

sdata[0] += sdata[2]
sdata[4] += sdata[6]

当 s = 4，只有 tid = 0 工作： 

sdata[0] += sdata[4]

最后 sdata[0] 就是整个 block 的 sum。

V1 为什么慢

Warp divergence 前面已经讲过：一个 warp 里的线程走不同分支，硬件需要分批执行。 V1 第一轮就是典型例子： 

thread 0 工作
thread 1 不工作
thread 2 工作
thread 3 不工作
...

同一个 warp 内交错工作，一半 lane 空转，这是严重 divergence。 另一个问题是控制条件里有： 

tid % (2 * s)

取模比简单比较、加法、位移更不友好。虽然编译器可能把某些 2 的幂取模优化成位运算， 但面试里重点应该放在 divergence，而不是只盯着 %。

V1 的问题是控制条件复杂，而且会导致严重 warp divergence。 % 这种条件判断形式也不如连续区间判断友好，但核心问题仍然是 divergence。

V2：为什么能减少 divergence

for (int s = 1; s < blockDim.x; s *= 2) {
    int index = 2 * s * tid;
    if (index < blockDim.x) {
        sdata[index] += sdata[index + s];
    }
    __syncthreads();
}

还是假设 blockDim = 8。 当 s = 1： 

index = 2 * tid

tid 0 → index 0
tid 1 → index 2
tid 2 → index 4
tid 3 → index 6
tid 4 → index 8，不工作

工作线程变成 tid = 0, 1, 2, 3，它们是连续的。 当 s = 2： 

index = 4 * tid

tid 0 → index 0
tid 1 → index 4
tid 2 → index 8，不工作

工作线程是 tid = 0, 1，也是连续的。 这比 V1 的“隔一个线程工作一个线程”好很多。

V2 的改善点是：在较大的 block 里，很多 warp 会变成整个 warp 都工作， 或者整个 warp 都不工作，而不是一个 warp 里交错工作。

V2 为什么会有 bank conflict

Bank conflict 前面在矩阵转置里已经详细解释过。 Shared memory 通常可以先按 32 个 bank 理解，float 下标大致用： 

bank = index % 32

V2 的访问是： 

sdata[index] += sdata[index + s];
index = 2 * s * tid;

当 s 变大时，相邻线程访问 shared memory 的地址间隔也变大： 

s = 1: index = 0, 2, 4, 6, ...
s = 2: index = 0, 4, 8, 12, ...
s = 4: index = 0, 8, 16, 24, ...

stride 为 2 时只用一半 bank；stride 为 4 时 bank 利用更差； stride 越大，越容易让多个线程落到同一批 bank。

V2 把工作线程变成连续线程，减少了 warp divergence； 但它通过 index = 2 * s * tid 访问 shared memory， 让相邻线程访问地址之间有 stride，可能产生 bank conflict。

V3：Sequential Addressing 为什么更好

V3 是经典 reduce 版本： 

for (int s = blockDim.x / 2; s > 0; s >>= 1) {
    if (tid < s) {
        sdata[tid] += sdata[tid + s];
    }
    __syncthreads();
}

核心变化是循环方向反过来： 

V1 / V2: s 从小到大
V3:      s 从大到小

以 blockDim = 8 为例，第一轮 s = 4， 工作线程是 tid = 0, 1, 2, 3： 

sdata[0] += sdata[4]
sdata[1] += sdata[5]
sdata[2] += sdata[6]
sdata[3] += sdata[7]

左边访问 sdata[0..3]，右边访问 sdata[4..7]， 都是连续地址。第二轮 s = 2： 

sdata[0] += sdata[2]
sdata[1] += sdata[3]

仍然是连续访问。第三轮 s = 1： 

sdata[0] += sdata[1]

所以 V3 同时改善两个问题：工作线程连续，减少 divergence； shared memory 访问连续，减少 bank conflict。

V3 把 stride 访问改成 sequential addressing。每一轮中， 活跃线程是连续的 tid < s，并且访问的 sdata[tid] 和 sdata[tid + s] 都是连续地址， 因此减少了 warp divergence 和 shared memory bank conflict。

V4：First Add During Load

V4 的关键是： 

int idx = blockIdx.x * (blockDim.x * 2) + threadIdx.x;
sdata[tid] = input[idx] + input[idx + blockDim.x];

V3 一开始每个 block 只处理 blockDim.x 个元素。 如果 blockDim = 256，一个 block 读 256 个数到 shared memory， 第一轮 s = 128 时，只有 128 个线程工作，另一半线程很快闲置。

V4 的想法是：既然第一轮一定是两两相加，为什么不在 load 的时候就加掉？ 让每个线程读两个数： 

sdata[tid] = input[idx] + input[idx + blockDim.x];

如果 blockDim = 256，一个 block 实际处理 512 个输入元素， 但 shared memory 里只存 256 个 partial sum。 这相当于一开始就完成了一轮： 

512 → 256
V4 的 first add during load 相当于在 global memory load 阶段就完成第一轮规约。 每个线程加载两个元素并相加，使每个 block 处理 2 * blockDim 个元素， 减少 block 数和后续规约开销，同时提高每个线程的工作量。

V4 的边界问题

学习版代码经常直接写： 

sdata[tid] = input[idx] + input[idx + blockDim.x];

但如果 n 不是刚好够，就可能越界。 工程里应该写成安全版： 

float sum = 0.0f;
if (idx < n) {
    sum += input[idx];
}
if (idx + blockDim.x < n) {
    sum += input[idx + blockDim.x];
}
sdata[tid] = sum;

这和前面向量加法里的 if (idx < n) 是同一类问题： kernel 启动线程数和真实数据规模不一定刚好相等，必须保护边界。

V5：为什么要 Unroll Last Warp

V5 的关键是把最后一个 warp 的规约单独拿出来： 

for (int s = blockDim.x / 2; s > 32; s >>= 1) {
    if (tid < s) {
        sdata[tid] += sdata[tid + s];
    }
    __syncthreads();
}

if (tid < 32) {
    warpReduce(sdata, tid);
}

当前面规约到 s <= 32 时，只剩 tid = 0 ~ 31 这 32 个线程在工作，刚好是一个 warp。 老 CUDA 代码经常利用 warp-synchronous programming： 一个 warp 内线程基本按同一条指令节奏执行，因此最后一个 warp 里可以省掉 block 级 __syncthreads()。

V5 会把最后几轮手动展开： 

sdata[tid] += sdata[tid + 32];
sdata[tid] += sdata[tid + 16];
sdata[tid] += sdata[tid + 8];
sdata[tid] += sdata[tid + 4];
sdata[tid] += sdata[tid + 2];
sdata[tid] += sdata[tid + 1];

这样减少循环和同步开销。

V5 里的 volatile 是什么

很多 V5 代码会写： 

__device__ void warpReduce(volatile float* sdata, int tid)

volatile 是新概念。这里它的作用是告诉编译器： 这块内存可能被其他执行单元改变，不要把它随便缓存到寄存器里复用， 每次都按共享内存读写来处理。

因为 warp 内线程通过 shared memory 交换数据： 

sdata[tid] += sdata[tid + 16];
sdata[tid] += sdata[tid + 8];
...

如果编译器过度缓存 sdata[tid] 的值，可能导致读不到其他线程刚写入的结果。 所以老式 warp-synchronous shared memory reduce 里会用 volatile 防止编译器优化破坏线程间通过 shared memory 交换数据。

现代 GPU 上 V5 的坑

这是面试加分点。早期 GPU 里，一个 warp 内 32 个线程基本可以认为 lockstep 执行。 但 Volta 之后引入了 Independent Thread Scheduling，也就是独立线程调度。 意味着同一个 warp 内的线程不再总是严格同步推进。

所以老代码里“反正一个 warp 内天然同步，我不用同步”的假设不再完全安全。 更现代的写法应该显式使用： 

__syncwarp();

或者更推荐用后面要讲的 warp shuffle。

V5 是经典 Mark Harris reduce 优化中的老式写法，它依赖 warp-synchronous programming。 早期架构中一个 warp 内线程 lockstep 执行，所以最后一个 warp 可以省略 __syncthreads()，并用 volatile 防止编译器缓存 shared memory。 但 Volta 之后有独立线程调度，这种隐式同步假设不再完全安全， 现代代码更推荐使用 __syncwarp() 或 warp shuffle。

V6：Warp Shuffle 是什么

Warp shuffle 是 Day 2 里最重要的新概念之一。它用的是： 

__shfl_down_sync(mask, val, offset)

它的作用是：让同一个 warp 内的线程直接从另一个线程拿寄存器里的值。 以前 warp 内线程交换数据需要： 

thread A 的寄存器 → shared memory → thread B 读取

有了 shuffle 之后可以变成： 

thread A 的寄存器 → thread B 的寄存器

少了 shared memory 这一步，也就减少了 shared memory 往返、bank conflict 和同步开销。

用 shuffle 做 warp 内 reduce

__device__ float warpReduceSum(float val) {
    for (int offset = 16; offset > 0; offset /= 2) {
        val += __shfl_down_sync(0xffffffff, val, offset);
    }
    return val;
}

假设一个 warp 里 32 个线程，每个线程有一个 val。 当 offset = 16： 

lane 0 拿 lane 16 的 val
lane 1 拿 lane 17 的 val
lane 2 拿 lane 18 的 val
...

前 16 个 lane 持有两两相加结果。 当 offset = 8，前 8 个 lane 继续累加； offset = 4, 2, 1 继续规约。 最后 lane 0 得到整个 warp 的 sum。

0xffffffff mask 是什么

__shfl_down_sync(0xffffffff, val, offset) 里的 0xffffffff 是 32 位全 1，表示 warp 内 32 个 lane 全部参与。 每一位代表一个 lane： 

bit 0  → lane 0
bit 1  → lane 1
...
bit 31 → lane 31

为什么现代 CUDA 要传 mask？因为 Volta 之后有独立线程调度， 硬件需要知道哪些线程应该参与这次 warp-level collective 操作。 _sync 版本要求显式传 mask，就是为了让参与线程集合明确。

Mask 用来指定 warp 中哪些 lane 参与本次 shuffle 操作。 0xffffffff 表示 32 个 lane 全部参与。 Volta 之后由于独立线程调度，warp-level primitive 需要显式 mask 来保证参与线程集合明确。

V6 为什么 shared memory 只开 32 个

很多 V6 代码只需要： 

__shared__ float sdata[32];

为什么不是 256？因为 V6 不再让所有线程都通过 shared memory 做规约。 它先让每个 warp 内部用 shuffle 规约。

假设 blockDim = 256： 

256 个线程 = 8 个 warp

每个 warp 得到一个局部 sum，所以只需要存 8 个 warp 的结果。 即使一个 block 有 1024 个线程： 

1024 / 32 = 32 个 warp

所以开 32 个 float 就够。 

每个线程读数据到寄存器 val
      ↓
每个 warp 内部用 shuffle 做 reduce
      ↓
每个 warp 的 lane 0 写一个结果到 shared memory
      ↓
第一个 warp 读取这些 warp 结果
      ↓
再用 shuffle 做一次 reduce
      ↓
block 的总和出来

V6 的完整逻辑拆解

int tid = threadIdx.x;
int lane = tid % 32;
int wid = tid / 32;

含义是： 

lane：当前线程在 warp 内的编号，0~31
wid： 当前线程属于 block 内第几个 warp

比如 blockDim = 256： 

tid 0~31   → wid = 0, lane = 0~31
tid 32~63  → wid = 1, lane = 0~31
tid 64~95  → wid = 2, lane = 0~31
...

第一步：每个线程读两个元素

这里仍然要用安全边界版本： 

float val = 0.0f;
if (idx < n) {
    val += input[idx];
}
if (idx + blockDim.x < n) {
    val += input[idx + blockDim.x];
}

第二步：每个 warp 内部 reduce

val = warpReduceSum(val);

执行完后，每个 warp 的 lane 0 持有该 warp 的 sum。

第三步：每个 warp 的 lane 0 写 shared memory

if (lane == 0) {
    sdata[wid] = val;
}

如果 blockDim = 256，就会写： 

sdata[0] = warp0_sum
sdata[1] = warp1_sum
...
sdata[7] = warp7_sum

第四步：同步

__syncthreads();

这里仍然需要同步，因为第一个 warp 接下来要读 sdata， 必须保证其他 warp 的 lane 0 已经写完。

第五步：第一个 warp 再 reduce 一次

val = (tid < blockDim.x / 32) ? sdata[lane] : 0;
if (wid == 0) {
    val = warpReduceSum(val);
}

如果 blockDim = 256，第一个 warp 的 lane 0 到 lane 7 读取 sdata[0] 到 sdata[7]，lane 8 到 lane 31 设为 0。 然后第一个 warp 再做一次 shuffle reduce。 最后： 

if (tid == 0) {
    output[blockIdx.x] = val;
}

一个 block 的 partial sum 就写出去了。

V6 为什么快

V6 的优势集中在几个点：

  1. Warp 内数据交换不走 shared memory。
  2. 不需要 volatile。
  3. Warp 内规约不需要 __syncthreads()。
  4. Shared memory 用量极少。
  5. Bank conflict 基本不是主要问题。
  6. 寄存器级通信延迟低。
Warp shuffle 允许同一个 warp 内线程直接交换寄存器中的值，不需要经过 shared memory， 因此减少了 shared memory round trip、bank conflict 和同步开销。 对于 warp 内规约，shuffle 通常是现代 CUDA 的首选方式。

Reduce 为什么是 memory bound

Memory bound 前面在转置里已经碰到过：瓶颈主要不是计算单元算不过来， 而是内存带宽不够。 Reduce 对每个 float 基本只做一次加法： 

读取 4 字节
做 1 次 add

算术强度很低，所以瓶颈通常是 global memory bandwidth，而不是 FP32 算力。 这也是为什么 reduce benchmark 通常用 GB/s 衡量，而不是 TFLOPS。

Reduce 是典型 memory-bound kernel。每个元素只参与一次加法，算术强度很低， 所以性能主要受 global memory bandwidth 影响。优化目标是减少不必要的访存和同步， 并尽可能提高带宽利用率，逼近显存带宽上限。

Benchmark 里的 bandwidth 怎么理解

float bandwidth = (n * sizeof(float)) / (ms / 1000.0f) / 1e9;

这里算的是有效输入读取带宽。严格来说还有写 partial output， 也就是 output[numBlocks]，但这个数据量远小于 input， 所以通常可以忽略。

如果 n = 16M： 

16M float ≈ 16 * 1024 * 1024 * 4 bytes ≈ 64MB

如果耗时 0.1ms，有效带宽大约就是： 

64MB / 0.1ms ≈ 640GB/s

Benchmark 常见坑

计时方法前面已经在转置里碰到过，核心还是 cudaEvent。 对 reduce 来说，不要只跑一次 kernel。更稳的做法是：

  1. 先 warmup 几次。
  2. 正式计时跑 100 次。
  3. 取平均时间。
  4. kernel 后检查 cudaGetLastError()。
  5. 确认没有把 cudaMemcpy 算进 kernel 时间。
for (int i = 0; i < 10; i++) {
    reduce_v6<<<grid, block>>>(d_in, d_out, n);
}

cudaEventRecord(start);
for (int i = 0; i < repeat; i++) {
    reduce_v6<<<grid, block>>>(d_in, d_out, n);
}
cudaEventRecord(stop);
cudaEventSynchronize(stop);

float ms = 0;
cudaEventElapsedTime(&ms, start, stop);
ms /= repeat;

如果 input 不是 2 的幂怎么办

很多教程默认 n 是 2 的幂，blockDim 也是 2 的幂。 比如： 

n = 16M
blockDim = 256

但真实工程不一定，所以边界判断必须写： 

float sum = 0.0f;
if (idx < n) {
    sum += input[idx];
}
if (idx + blockDim.x < n) {
    sum += input[idx + blockDim.x];
}

这样不管 n 是不是 2 的幂，都不会越界。 另外，blockDim 仍然建议选择 128、256、512、1024 这类 2 的幂， 因为 s >>= 1 的 reduce 循环天然适合这种大小。

Grid-stride loop 是什么

Grid-stride loop 是 Reduce 进阶里非常常见的新概念。 普通写法是每个线程只处理一个元素： 

int idx = blockIdx.x * blockDim.x + threadIdx.x;
if (idx < n) {
    ...
}

Grid-stride loop 则让一个线程处理多个元素： 

for (int i = blockIdx.x * blockDim.x + threadIdx.x;
     i < n;
     i += blockDim.x * gridDim.x) {
    sum += input[i];
}

意思是：线程处理完自己的第一个元素后，跳过整个 grid 的线程数， 继续处理下一个属于自己的元素。

它的好处是：

  1. n 很大时，不需要启动特别多 block。
  2. 每个线程可以处理多个元素，提高工作量。
  3. 更容易处理任意大小输入。
  4. 可以减少 partial sum 数量。

现代 reduce 常常结合 first add during load 写成： 

float sum = 0.0f;
for (int i = blockIdx.x * blockDim.x * 2 + threadIdx.x;
     i < n;
     i += blockDim.x * gridDim.x * 2) {
    sum += input[i];
    if (i + blockDim.x < n) {
        sum += input[i + blockDim.x];
    }
}

然后再对每个线程的 sum 做 block 内 reduce。 这里 sum 是线程私有局部变量，前面讲过，它通常会放在 register 里。

不要背 6 个版本，要背演进故事线

版本 核心写法 解决了什么 还有什么问题
V1 tid % (2*s) == 0 最基础树形规约 Warp divergence，取模开销
V2 index = 2*s*tid 工作线程更连续，减少 divergence Stride shared memory 访问，bank conflict
V3 s = blockDim/2 → 1 Sequential addressing，减少 bank conflict 一开始一半线程很快闲置
V4 Load 两个元素先相加 Block 数减半，提高线程工作量 最后 warp 仍有同步和循环开销
V5 Unroll last warp 减少最后 warp 的同步和循环 老式 warp-synchronous，现代 GPU 有坑
V6 Warp shuffle 寄存器级规约，减少 shared memory 和同步 需要理解 mask、lane、wid、边界

面试里怎么讲 Reduce 演进

我会先从 block 内 shared memory reduce 开始。最朴素的 V1 使用 interleaved addressing， 通过 tid % (2*s) == 0 选择工作线程，但这会导致严重 warp divergence， 而且取模开销也不友好。V2 把工作线程变成连续线程，减少 divergence， 但由于 index = 2*s*tid 产生 stride shared memory 访问，会引入 bank conflict。 V3 改成 sequential addressing，从 blockDim/2 开始向下规约， 让活跃线程连续访问 sdata[tid] 和 sdata[tid+s]， 减少 bank conflict。V4 进一步在 load 阶段让每个线程读取两个元素并相加， 相当于提前做一轮 reduce，减少 block 数。V5 把最后一个 warp 手动展开， 减少 __syncthreads() 和循环开销，但它依赖旧式 warp-synchronous programming。 现代 GPU 更推荐 V6，用 __shfl_down_sync 在 warp 内直接做寄存器级规约， 然后每个 warp 写一个 partial sum 到 shared memory，最后由第一个 warp 再规约一次。 Reduce 本质是 memory-bound，所以最终优化目标是提高 global memory bandwidth 利用率。

老师学习版代码里的几个坑

V4 / V5 / V6 都要加边界判断

不要直接写： 

input[idx] + input[idx + blockDim.x]

要写： 

float val = 0.0f;
if (idx < n) {
    val += input[idx];
}
if (idx + blockDim.x < n) {
    val += input[idx + blockDim.x];
}

BlockDim 最好是 2 的幂

比如 128、256、512、1024。因为 reduce 循环里的 s >>= 1 默认适合 2 的幂。

一个 kernel 只得到 partial sum

不要以为： 

output[blockIdx.x] = sdata[0];

之后就结束了。如果 grid 有 1024 个 block，输出就是 1024 个 partial sum， 还要继续 reduce。

V5 老写法不要当现代最佳实践

V5 是为了理解历史优化路线。现代代码更推荐 warp shuffle， 或者至少显式使用 __syncwarp()。

Day 2 最短理解版本

Reduce 优化的本质，是让线程先在寄存器里多干一点，再用 shared memory 做 block 内规约， 避免 warp divergence 和 bank conflict，最后用 warp shuffle 替代 shared memory 完成 warp 内通信， 尽可能减少同步和访存开销，让性能接近显存带宽上限。 
V1 解决能不能并行
V2 解决线程分歧
V3 解决 shared memory 访问
V4 解决线程工作量不足
V5 解决最后 warp 同步开销
V6 解决 warp 内通信效率

把执行模型和内存层级合起来

第一部分讲的是：thread 组成 block，block 被调度到 SM，SM 把线程拆成 warp 执行。 第二部分讲的是：这些 warp 在执行指令时，数据会从 register、shared memory/L1、L2、global memory 这些层级里来回流动。第三部分讲的是：真正写 kernel 时，性能差异经常来自 global memory 是否连续访问、 shared memory 是否真的带来数据复用，以及 shared memory 内部有没有 bank conflict。 第四部分讲的是：像 reduce 这种看起来串行的汇聚问题，如何通过树形规约、first add during load、 warp shuffle 和多轮 partial sum 变成 GPU 适合的并行问题。

CUDA 内存优化的本质，就是用更快、更小、更近的内存，去减少更慢、更大、更远的内存访问。

所以以后看到 tiling、coalesced access、occupancy，不要把它们当成三块零散知识。 它们其实是同一套逻辑：Global Memory 很慢，所以要减少访问次数；如果数据会重复用， 就搬到 Shared Memory；如果是线程自己的临时变量，就尽量放 Register； 如果必须访问 Global Memory，就让相邻线程访问连续地址；如果访问延迟不可避免， 就提高 Occupancy，让更多 warp 帮你隐藏延迟。
