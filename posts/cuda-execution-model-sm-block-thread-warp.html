CUDA 编程模型入门：SM、Block、Thread 和 Warp 到底怎么配合

2026.05.18 CUDA GPU SM Warp 性能优化

CUDA 编程模型的核心，就是把大量线程分组，然后让 GPU 的多个 SM 去并行执行这些线程组。 如果先不用术语，可以把它理解成一座工厂：GPU 是整座工厂，SM 是工厂里的车间， Block 是分配给车间的一批任务，Thread 是真正干活的工人，Warp 是 32 个工人组成的固定小队。 

一张 GPU = 一个工厂
SM       = 工厂里的车间
Block    = 分配给车间的一批任务
Thread   = 干活的工人
Warp     = 32 个工人组成的固定小队

你现在先记住这句话：CUDA 不是把一个大任务丢给一个“大核心”，而是把任务拆成很多 block， 再把这些 block 调度到很多 SM 上执行；每个 SM 内部又把线程按 warp 来执行。 后面所有概念都围绕这句话展开。

SM 是什么

SM 全称是 Streaming Multiprocessor，可以理解为 GPU 里真正干活的计算单元。 CPU 通常说 8 核、16 核；GPU 的结构不是一个超大的核心，而是由几十个甚至上百个 SM 组成。 每个 SM 里面有 CUDA Core、寄存器、共享内存、调度器等硬件资源。

所以你写 CUDA kernel 的时候，本质上是让很多线程被组织成 block， 再让 GPU 把这些 block 分配到不同 SM 上执行。你不直接调度 CUDA Core， 你真正需要重点理解的是 SM，因为 block 是被分配到 SM 上的。

3080 Ti 有多少个 SM

RTX 3080 Ti 使用 GA102 芯片，一共有 80 个 SM。 每个 SM 里有 128 个 FP32 CUDA Core，所以总 FP32 CUDA Core 数量是： 

80 × 128 = 10240 个 CUDA Core

但这里要注意：CUDA Core 不是你写代码时直接调度的单位。 你启动 kernel 时配置的是 grid 和 block，硬件调度时以 block 为单位分配到 SM。 理解了 SM，才算真正开始理解 CUDA 的执行模型。

Grid、Block、Thread 是什么关系

CUDA 里启动一个 kernel，一般长这样： 

kernel<<<gridDim, blockDim>>>(...);

比如： 

myKernel<<<4, 256>>>(...);

这句话的意思是：启动 4 个 block，每个 block 有 256 个 thread， 所以总线程数是： 

4 × 256 = 1024 个 thread

结构可以画成这样： 

Grid
 ├── Block 0
 │    ├── Thread 0
 │    ├── Thread 1
 │    ├── ...
 │    └── Thread 255
 │
 ├── Block 1
 │    ├── Thread 0
 │    ├── Thread 1
 │    ├── ...
 │
 ├── Block 2
 │
 └── Block 3

所以 CUDA 的三级结构是： 

Grid  →  Block  →  Thread
整个任务 → 任务小组 → 具体线程

一个重要细节是：一个 block 会被放到一个 SM 上执行，不会被拆到多个 SM 上。 这也是为什么同一个 block 内的线程可以使用 shared memory，也可以用 __syncthreads() 做同步；不同 block 之间不能直接同步，也不能直接共享 block 内的 shared memory。

CUDA 内置变量：blockIdx、threadIdx、blockDim、gridDim

这些变量是 CUDA 自动给你的。你不用自己定义，kernel 里可以直接用。

变量 含义 myKernel<<<4, 256>>> 里的值
threadIdx.x 当前线程在自己 block 里的编号 0 ~ 255
blockIdx.x 当前 block 在整个 grid 里的编号 0 ~ 3
blockDim.x 每个 block 有多少个线程 256
gridDim.x 整个 grid 有多少个 block 4

threadIdx 只是在当前 block 内部编号，不是全局编号。 所以 Block 0 里有 threadIdx.x = 0， Block 1 里也有 threadIdx.x = 0， Block 2 里也有 threadIdx.x = 0。 每个 block 内部都会从 0 重新编号。

怎么算一个线程的全局编号

这是 CUDA 最核心的公式之一： 

int idx = blockIdx.x * blockDim.x + threadIdx.x;

还是看这个启动配置： 

myKernel<<<4, 256>>>(...);

每个 block 里有 256 个线程，所以全局线程编号是： 

Block 0:
idx = 0 * 256 + threadIdx.x
idx = 0 ~ 255

Block 1:
idx = 1 * 256 + threadIdx.x
idx = 256 ~ 511

Block 2:
idx = 2 * 256 + threadIdx.x
idx = 512 ~ 767

Block 3:
idx = 3 * 256 + threadIdx.x
idx = 768 ~ 1023

所以这行代码的作用，就是把 block 内部编号转换成整个 grid 里的全局线程编号。 以后你写 CUDA kernel，十有八九都会看到它。

Block 是怎么分配到 SM 上的

当你启动： 

myKernel<<<100, 256>>>(...);

表示你创建了 100 个 block，每个 block 256 个 thread。 这些 block 会被 GPU 的硬件调度器分配到 SM 上执行。

如果你的 GPU 是 RTX 3080 Ti，有 80 个 SM，那么大概可以想成： 

SM0  分到一些 block
SM1  分到一些 block
SM2  分到一些 block
...
SM79 分到一些 block

但你不能手动指定某个 block 去某个 SM。CUDA 的调度规则可以粗略理解为： 哪个 SM 有空，哪个 SM 就拿下一个 block 执行。 

80 个窗口办理业务
100 个客户排队
哪个窗口空了，下一个客户就过去

窗口 = SM
客户 = Block

如果 block 数很多，SM 执行完当前驻留的一批 block 之后，会继续拿下一批 block。 这就是 GPU 能同时处理大量任务的基础。

一个 SM 能同时跑多少个 block

这个问题不能只看 GPU 型号，还要看你的 kernel 用了多少资源。 一个 SM 上可以同时驻留多个 block，这叫 resident blocks，也可以理解成同一个 SM 上同时挂着多个 block。

但一个 SM 能同时放几个 block，受这些因素限制：

  1. 每个 block 有多少线程。
  2. 每个线程用了多少寄存器。
  3. 每个 block 用了多少 shared memory。
  4. GPU 架构本身的硬件上限。

以 3080 Ti / Ampere 架构为例，一个 SM 的常见上限是： 

最多 1536 个 resident threads
最多 48 个 resident warps
最多 16 个 resident blocks

这些只是理论上限。你的 kernel 如果用了很多 shared memory 或寄存器，实际能驻留的 block 数会更少。

例子 1：每个 block 256 个线程

myKernel<<<..., 256>>>(...);

一个 block 有 256 个线程，也就是 8 个 warp。 如果只看线程数，一个 SM 最多 1536 个线程： 

1536 / 256 = 6 个 block

所以理想情况下，一个 SM 可以同时放 6 个 256-thread block。

例子 2：每个 block 1024 个线程

myKernel<<<..., 1024>>>(...);

一个 block 有 1024 个线程，也就是 32 个 warp。 只看线程数： 

1536 / 1024 = 1 个 block

虽然还剩 512 个线程容量，但不够再放一个 1024-thread block。 所以一个 SM 同时只能放 1 个这种 block。这也是为什么 block size 不是越大越好。

例子 3：每个 block 128 个线程

myKernel<<<..., 128>>>(...);

一个 block 有 128 个线程，也就是 4 个 warp。只看线程数： 

1536 / 128 = 12 个 block

所以一个 SM 理论上可以同时放 12 个 128-thread block。 但如果 kernel 使用了很多寄存器或 shared memory，实际数量可能低于 12。

Warp 是什么

Warp 是 CUDA 真正执行线程的基本单位。一个 warp 固定是 32 个线程。 你写代码的时候，感觉每个 thread 是独立执行的；但硬件执行的时候， 不是一个 thread 一个 thread 执行，而是 32 个 thread 组成一个 warp， 一起执行同一条指令。

这种执行模型叫 SIMT，Single Instruction, Multiple Threads。 可以理解成：一条指令，多个线程一起干。

比如这个最常见的向量加法 kernel： 

__global__ void add(float* a, float* b, float* c) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    c[idx] = a[idx] + b[idx];
}

假设一个 warp 里的 32 个线程编号是 thread 0 到 thread 31。 它们会一起执行： 

c[idx] = a[idx] + b[idx];

只是每个线程的 idx 不同： 

thread 0  处理 c[0]  = a[0]  + b[0]
thread 1  处理 c[1]  = a[1]  + b[1]
thread 2  处理 c[2]  = a[2]  + b[2]
...
thread 31 处理 c[31] = a[31] + b[31]

所以看起来是 32 个线程各干各的，实际上是硬件把它们作为一组来执行。

为什么 Warp 是理解性能问题的起点

GPU 性能问题很多都和 warp 有关。最重要的三个概念是：

  1. Warp divergence：分支发散。
  2. Memory coalescing：内存合并访问。
  3. Occupancy：SM 上有多少 warp 能同时驻留。

分支发散：warp divergence

看这个代码： 

if (threadIdx.x % 2 == 0) {
    // 偶数线程做 A
} else {
    // 奇数线程做 B
}

一个 warp 里有 32 个线程，其中 16 个线程走 if，16 个线程走 else。 问题是：一个 warp 里的线程应该一起执行同一条指令。 但现在一半要执行 A，一半要执行 B。

GPU 只能先让偶数线程执行 A，奇数线程暂时屏蔽； 再让奇数线程执行 B，偶数线程暂时屏蔽。 原来一次能干完的事情，现在变成两次，这就叫 warp divergence。

同一个 warp 里的线程走了不同分支，执行会被拆开，性能就会下降。

内存合并访问：memory coalescing

GPU 访问 global memory 的时候，很喜欢相邻线程访问相邻内存。 比如： 

c[idx] = a[idx] + b[idx];

一个 warp 里的线程访问： 

a[0], a[1], a[2], ..., a[31]

这些地址是连续的。硬件可以把这些访问合并成更少的内存事务， 这叫 coalesced memory access，也就是内存合并访问，性能很好。

但如果你这样访问： 

c[idx] = a[idx * 32] + b[idx * 32];

一个 warp 里的线程访问： 

a[0], a[32], a[64], a[96], ...

地址跳得很远，GPU 就很难合并访问，global memory 访问效率会下降。 所以你以后优化 CUDA kernel，经常会听到一句话：

让相邻线程访问相邻内存。

这句话的根源就是 warp。

Occupancy：占用率

Occupancy 可以粗略理解为：一个 SM 上同时挂了多少个 warp。 如果一个 SM 上只有很少的 warp，当某个 warp 等 global memory 的时候，SM 可能就没活干了。

如果一个 SM 上有很多 warp，那么： 

warp A 等内存
SM 切去执行 warp B

warp B 等内存
SM 切去执行 warp C

warp C 等内存
SM 再切去执行别的 warp

这样可以把等待时间隐藏掉。GPU 不是靠“单个线程很快”取胜， 而是靠大量 warp 切换来隐藏延迟。

但也要注意，occupancy 不是越高越好。它是重要指标，但不是唯一目标。 如果为了提高 occupancy 牺牲了数据复用、增加了访存或让寄存器溢出，性能也可能变差。 对入门阶段来说，先理解“足够多的 warp 可以隐藏延迟”就够了。

Block 和 Warp 的关系

一个 block 里面的线程，会被拆成多个 warp。公式是： 

warp 数 = ceil(blockDim.x / 32)

常见 block size 对应的 warp 数：

blockDim.x Warp 数
128 128 / 32 = 4
256 256 / 32 = 8
512 512 / 32 = 16
1024 1024 / 32 = 32

如果 blockDim.x 不是 32 的倍数，比如 100： 

ceil(100 / 32) = 4 个 warp

但是最后一个 warp 只有 4 个有效线程，剩下 28 个线程位置浪费。 所以 CUDA 里 block size 通常会设成 32 的倍数，比如 128、256、512。

完整例子：向量加法

假设你有两个数组： 

a = [1, 2, 3, 4, ...]
b = [10, 20, 30, 40, ...]

你想算： 

c[i] = a[i] + b[i]

CUDA kernel 可以这样写： 

__global__ void vectorAdd(float* a, float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

启动代码： 

int blockSize = 256;
int gridSize = (n + blockSize - 1) / blockSize;

vectorAdd<<<gridSize, blockSize>>>(a, b, c, n);

这里 blockSize = 256，表示每个 block 256 个线程。 gridSize 的计算是为了让总线程数覆盖 n 个元素： 

int gridSize = (n + blockSize - 1) / blockSize;

比如： 

n = 1000
blockSize = 256
gridSize = (1000 + 256 - 1) / 256 = 4

总线程数是： 

4 × 256 = 1024

但我们只需要处理 1000 个元素，所以最后 24 个线程会超出范围。 因此 kernel 里需要： 

if (idx < n)

这句判断是防止越界访问。几乎所有处理数组的 CUDA kernel 都会有类似保护。

CUDA 执行流程应该怎么理解

当你写： 

vectorAdd<<<gridSize, blockSize>>>(a, b, c, n);

实际发生的是：

  1. CPU 发起 kernel 启动请求。
  2. GPU 创建一个 grid。
  3. Grid 里包含很多 block。
  4. GPU 把 block 分配给各个 SM。
  5. 每个 SM 接收一个或多个 block。
  6. Block 内部线程被拆成 warp。
  7. Warp 以 32 个线程为一组执行指令。
  8. 每个 thread 根据自己的 idx 处理对应数据。

可以画成： 

GPU
├── SM 0
│   ├── Block 0
│   │   ├── Warp 0: Thread 0  ~ 31
│   │   ├── Warp 1: Thread 32 ~ 63
│   │   └── ...
│   └── Block 80
│
├── SM 1
│   ├── Block 1
│   └── Block 81
│
├── SM 2
│   ├── Block 2
│   └── Block 82
│
...
└── SM 79
    ├── Block 79
    └── Block 159

如果 block 数很多，SM 执行完一批 block 后，会继续拿下一批 block。 所以 block 数通常要足够多，否则你可能根本喂不满所有 SM。

面试里怎么说

如果面试官问 CUDA 的执行模型，可以这样回答：

CUDA 的执行模型是 Grid、Block、Thread 三级结构。一个 kernel 启动后会生成一个 Grid， Grid 由多个 Block 组成，每个 Block 又包含多个 Thread。Block 是调度到 SM 上的基本单位， GPU 会把不同的 Block 分配到不同 SM 上执行。一个 SM 可以同时驻留多个 Block， 但数量受线程数、寄存器、shared memory 和硬件上限影响。Thread 在硬件上不是单独调度执行， 而是以 Warp 为单位执行，一个 Warp 固定 32 个线程，采用 SIMT 模型。很多 CUDA 性能问题， 比如分支发散、内存合并访问、occupancy，本质上都要从 Warp 的执行方式来理解。

这个回答已经够专业了。再往深处追，就是具体架构、寄存器压力、shared memory、memory hierarchy、 Nsight profile 和具体 kernel 优化策略。

最应该记住的几个公式

全局线程 id

int idx = blockIdx.x * blockDim.x + threadIdx.x;

gridSize 计算

int gridSize = (n + blockSize - 1) / blockSize;

一个 block 里有多少 warp

warp 数 = ceil(blockDim.x / 32)

256 线程 block 对应多少 warp

256 / 32 = 8 个 warp

最后用一句话总结

CUDA 的核心不是“我启动了很多线程”这么简单，而是：

你启动很多 block；GPU 把 block 分配给不同 SM；SM 把 block 里的线程拆成 warp； warp 以 32 个线程为单位执行；性能好不好，主要看你的 warp 有没有高效执行、 内存访问有没有连续、SM 有没有足够多的活干。

理解到这个程度，就可以继续往 CUDA 性能优化走了。下一步不是背更多术语， 而是拿一个真实 kernel，用 Nsight 看它的 occupancy、memory throughput 和 warp 执行效率。
